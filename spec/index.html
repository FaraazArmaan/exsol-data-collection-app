1. The problem

Small and mid-sized businesses in India and similar markets prepare product catalogs for online sale across multiple platforms (Amazon, Flipkart, Meta Catalog, WhatsApp Business, Swiggy, Zomato, and custom storefronts). Today this is done by hand: a single spreadsheet shared between the business owner and a small team, manually re-shaped each time a marketplace asks for its own columns. Stock counts go stale because there is no shared, multi-user system for the storekeeper to record what was sold or received. Marketplace uploads are error-prone.

2. The solution

ExSol Data Collection is a multi-tenant web application that gives an Admin (the operator running the service) a workspace for each Client (a business) and lets the client's team collaboratively maintain product data and stock counts. The application is the canonical source of truth; downstream systems consume from it via exports and a forthcoming internal API.

Topology

The app is the hub in a hub-and-spoke design. It does not call marketplaces or external storefronts; it produces files (CSV, XLSX, Meta-schema CSV) that they consume, and a future Internal Website / ERP that reads from ExSol fans the data out to consumer-facing sites (booking pages, catalog pages, the custom ecommerce front).

4. Core workflows

  1. Client onboarding. Admin signs in, enters the Client's name and Primary user's email. The system creates a Workspace, generates a 12-character access key (shown once), and emails-or-shares the key out-of-band.
  2. Primary sign-in. Primary signs in with their Google account; the email match links the account on first sign-in. They land on their workspace's product dashboard.
  3. Product CRUD. Primary or Manager creates products with canonical fields (SKU, name, price, stock, dimensions, GST/HSN) and optionally enables per-marketplace overlays (Amazon, Flipkart, Meta, WA, Rakuten, AliExpress, Swiggy, Zomato).
  4. Stock ledger. Anyone with stock-write permission records movements (purchase, sale, damage, recount, manual adjust). Current count is derived from SUM(deltas).
  5. Exports. Primary or Manager exports the catalog as comprehensive XLSX, comprehensive CSV, or Meta-Catalog-schema CSV. Small exports run synchronously; large exports run as background jobs.
  6. Backups. Primary triggers per-workspace ZIP backups on demand. Admin's nightly scheduled function takes a full-system backup.
  7. Admin impersonation. When a Client needs help, Admin enters the workspace's access key (15-minute unlock), then begins an impersonation session with a written reason. A red banner pins on every page for 30 minutes; every write is recorded with both the real actor (Admin) and the on-behalf-of user (the Client member).

5. Tech stack snapshot

  • Frontend: Vanilla HTML / CSS / ES-module JavaScript, served as static files by Netlify.
  • Backend: Netlify Functions in TypeScript.
  • Database: Neon Postgres with Row-Level Security on every workspace-scoped table.
  • File storage: Netlify Blobs (product-images store; ≤5 MB per file via multipart POST).
  • Auth: Google Identity Services (primary) + email + password with Argon2id (fallback). JWT in HTTP-only cookies, 15-minute access + 30-day refresh with rotation.
