2. Key relationships

  • A Workspace has exactly one Primary User (referenced by workspaces.primary_user_id) and zero-to-many additional Workspace Memberships.
  • A User can be a member of many Workspaces, each potentially with a different role. The Admin user has no Workspace Memberships — Admin authority is at the system level via users.is_admin.
  • A Product has a canonical core (one row in products) and zero-to-many Marketplace Overlays (rows in product_marketplace_fields keyed by marketplace).
  • A Stock Movement never updates a Product directly. The products.stock_count column is derived: a Postgres trigger on stock_movements INSERT applies the delta to the product. The ledger is append-only.
  • An Audit Event records the real actor in actor_user_id and (when impersonation is active) the on-behalf-of user in on_behalf_of. Both columns reference Users.

3. Tenant isolation

Every workspace-scoped table has a workspace_id column and Row-Level Security enabled. A request opens a transaction, sets the GUCs app.current_user_id, app.current_workspace_id, and app.is_admin, runs the query, and commits. Postgres enforces the policy server-side regardless of what the application code's WHERE clause says. 

USING (
  is_admin_context()
  OR workspace_id = current_workspace_id()
)

Three context flavors exist in code: withTenantContext (user + workspace), withAdminContext (admin, cross-workspace), and withUserContext (user only, no workspace selected — used by /api/me to list a user's memberships).

The single most security-critical pair of modules is tenancyContext and permissionPolicy. The first prevents data leakage at the database layer; the second prevents privilege escalation at the application layer. Both are in the test tier.

4. The marketplace overlay pattern

Different marketplaces require different fields for the same product (Amazon wants 200+ attributes, Meta wants 10, Swiggy wants menu-shaped data). To support this without a 500-column products table or an Entity-Attribute-Value monster, we use a core + overlay shape:

  • The products table holds canonical fields every product has (SKU, name, price, stock, dimensions, GST/HSN, etc.).
  • The product_marketplace_fields table holds per-marketplace JSONB blobs. One row per (product, marketplace) pair. Each row has an enabled flag and a fields JSONB.
  • To list products enabled for a given marketplace, join on product_marketplace_fields with enabled = true.

Adding a new marketplace is a code change (extend the Marketplace enum) — not a schema migration. v1 supports amazon, flipkart, meta, wa, rakuten, aliexpress, swiggy, zomato.

5. The stock ledger pattern

Stock is a function of history, not a fact you overwrite. Every change is a row in stock_movements with a positive or negative integer delta, a reason (purchase, sale, damage, recount, manual_adjust), a source (manual, csv, recount), an actor, optional impersonation marker, and a timestamp. The current count derives from SUM(delta).

A "recount to absolute" is mechanically a movement with reason = recount and a delta computed to reach the target. The ledger never lies about history.
