Read this first

The honest short answer

ExSol Booking is a functional v1 appointment and reservation engine with a customer-facing flow on the shared public storefront. It supports Booking Setup, Workforce-backed availability, multi-service visits, policy snapshots, visit history, guest booking, front-desk operations, and independently published Ordering and Booking storefront features. It is not yet a complete production booking product.

Safe present-day promise: a business can configure services and capacity, offer policy-controlled pay-at-venue bookings, use Workforce shifts and leave for staff availability, and operate visits from the workspace. Still not ready to promise: verified online payments, reliable queued reminders, complete privacy operations, all client-facing capacity wording, and real-browser release verification of every public-storefront state.

This document calls a feature good when its core rule is enforced on the server and has meaningful tests, repair when the UI or API can violate its intended business rule, and missing when a normal production operator would expect it but the system has no complete implementation.

First principles

What a booking module really does

A booking module is a capacity-allocation system. A customer asks for a service at a time. The system must assign a resource that can perform it, reserve that capacity once, communicate the outcome, and preserve a trustworthy history for operations and finance.

Input

Demand

Customer, selected service, desired time, staff preference, contact consent, payment choice.

Decision

Availability

Business hours, resource hours, holidays, leave, service duration, buffers, lead time, existing reservations.

Outcome

Reservation

One durable appointment record, a lifecycle state, a payment state, notifications, and an audit trail.

Important distinction: a slot is usually not stored as a database row. In ExSol it is calculated when requested. The durable thing is the appointment time range. That is a valid and scalable design when the database protects against overlaps.
Client-facing language rule: customers and workspace operators never see the generic word “resource.” It remains an internal database/API term meaning scarce capacity. The client configures real-world things such as team members, doctors, rooms, stations, vehicles, equipment, or availability through Booking Setup.
What customers see instead: Booking Setup

The first visit to Booking Setup launches guided onboarding. It asks plain-language business questions, creates the internal capacity records and reservation rules, then hides technical terminology permanently.

Who do customers book with?
Specific team member · Any available team member · Nobody specific, just a time/place
What can be booked?
Appointments · Rooms/spaces · Equipment/assets · A combination
Does each booking need anything besides a team member?
No · A room/space · Equipment · Both
Where does availability come from?
Workforce shifts and leave · Manual business hours for now
Connect services in plain language.
“This service can be performed by…” · “This service needs…” · “This booking uses…”

The current workspace navigation is deliberately fixed: Calendar, Bookings, Services, Booking Setup, Booking Rules, and Settings. The old Resources route redirects to Booking Setup. Setup already saves the labels needed for a solo consultant (Your Availability), salon (Stylists and Stations), clinic (Doctors, Rooms, and Equipment), or equipment rental business (Assets). Applying those labels consistently, rather than adding generic capacity-management tabs, is remaining presentation work.

Engineering assessment

What is good, bad, and missing

Good

The database owns “never double book”

The EXCLUDE USING gist constraint is the right authority. Two concurrent requests cannot both create overlapping live appointments for one resource. This is materially better than trusting a browser’s displayed slot list.

Good

Scheduling logic is pure and tested

Time-zone conversion, schedule intersection, buffers, and least-busy assignment live in small reusable functions. This makes the highest-risk calculations testable without HTTP or a browser.

Good

Tenant isolation is deliberate

Vendor queries scope records by bucket_id/clientId; private guest management uses an unguessable token; settings and internal capacity records belong to one client.

Good

Access fits the platform

The module is registered correctly, enabled through a product, has dedicated navigation, uses platform permission keys, and includes L1 Owner bypass in server and client gates.

Good

Booking writes use a shared reservation path

Public create, vendor manual create, guest reschedule, and vendor reschedule now validate through the shared reservation and visit helpers before changing durable records. The database exclusion constraint remains the final concurrency guard.

Repair

Configured capacity labels are not consumed everywhere

Booking Setup stores labels such as Doctors, Stylists, Rooms, and Assets, but the public picker and some workspace service copy still expose generic or staff-specific wording. Add one label resolver and use it at every display boundary.

Repair

Payment is a false-complete surface

Paid service modes have a database enum and a stub response, not Razorpay order creation, checkout, verified state reconciliation, idempotency, refund flow, or payout reconciliation. Keeping the UI gate off is correct; do not turn it on early.

Repair

Contracts and permissions still have small local duplication

The module now owns booking.css, but shared/api.ts combines browser wrappers with UI contracts and the full permission list is local to the route mount. Extract the permission list now; extract shared types only when a second consumer would otherwise duplicate them.

Missing

Notification reliability

No complete reminder, cancellation, reschedule, staff alert, retry, delivery-status, SMS/WhatsApp, or customer preference system exists. A booking product without reminders will generate avoidable no-shows.

Partial

Operational audit and recovery

Visit events now preserve actor, source, previous/new state, and reason. Remaining work is optimistic concurrency, required reasons for staff overrides, and a payment-to-booking reconciliation view.

Missing

Business breadth

No multi-location capacity model, resource skills, recurring appointments, waitlist, packages/memberships, coupons, intake forms, deposits/refunds, external calendars, or customer self-service authentication.

Missing

Measurement

Analytics and Finance consume simple booking outcomes/revenue, but Booking lacks its own dashboard for occupancy, cancellation/no-show rate, lead time, resource utilization, conversion, and cohort retention.

Priority rule: do not add attractive features such as memberships or calendar sync before the current shared scheduling kernel is extended to them. A reliable reservation service is the prerequisite for every later capability.
Connection map

How the codebase connects a click to a database row

The browser never talks directly to Postgres. It renders React components, calls a same-origin API route, the Netlify Function authenticates and validates, then it reads or writes Neon Postgres. This separation prevents users from bypassing the interface.

  POST /api/booking-public/:slug/create

Route: src/lib/router.tsx maps public /storefront/:slug/Book and vendor /c/:slug/booking/* URLs to components. Older public booking URLs redirect to the shared storefront shell.
UI: public and vendor React components call typed wrapper functions in src/modules/booking/shared/api.ts.
Function: each flat netlify/functions/booking-*.ts file declares its HTTP route and method, parses input, authorizes, and returns JSON.
Domain logic: pure scheduling and state-transition rules live in src/modules/booking/lib/; server-only reservation and visit orchestration live in netlify/functions/_booking-reservation.ts and _booking-visits.ts. All booking write paths use that shared validation.
Storage: database tables record the durable truth; the query result becomes API JSON; React stores only temporary display state.
Why both client and server permissions?

The client gate keeps a user from seeing a page they cannot use. The server gate is the security boundary: a user can always forge an HTTP request from developer tools, so every mutation must verify session, product enablement, owner bypass, and permission on the server. ExSol’s requireBooking wrapper is built for this.

Why store money in cents and time in UTC?

Decimal floating-point arithmetic can produce errors such as 0.1 + 0.2. Integer cents avoid that. Local times are ambiguous around daylight-saving changes; UTC instants are not. Store tstzrange in UTC, but calculate and display business hours in the tenant’s IANA timezone such as Asia/Kolkata.

What should never be the source of truth?

A React state variable, browser local storage, a cached availability response, an email, or a payment-provider redirect must not be the appointment truth. They are conveniences. PostgreSQL appointment records plus a verified payment/event ledger are the truth.

Platform map

Which ExSol modules Booking needs, and who needs Booking

Booking consumes

Dependencies

  • Platform identity and access: tenant, user session, permission matrix, user nodes.
  • Payments: provider abstraction, order/payment/refund records, signed webhooks.
  • Email: transactional templates and delivery outbox.
  • Workforce: staff identities, shifts, approved leave, and employment status. When configured, Booking reads Workforce as the authoritative staff-availability source through the internal capacity link.
  • CRM: canonical customer profile, notes, consent, timeline.
  • Branding: public storefront identity, logo, colors, domains.
  • Files: intake attachments, consent forms, before/after media, only if business needs it.
Booking produces

Downstream data

  • Analytics: booking outcomes are already queried by analytics-bookings.ts.
  • Finance: confirmed/completed booking price is already included as a revenue channel; payment settlement must eventually refine this.
  • Marketing: current ROI links completed bookings by email; future segmentation needs canonical customer IDs and consent.
  • CRM: appointment timeline, visit history, no-show/cancellation facts, preferences.
  • Workforce: utilization, commissions, attendance versus scheduled work.
  • POS/Orders: check-in and pay-at-venue collection should refer to the same appointment/payment record, not duplicate a sale.
Ownership rule: Booking owns appointment facts. Payments will own provider transactions. CRM should own rich customer profile. Workforce owns employment and shifts. Finance owns accounting treatment. Integration means reference IDs and events, not copying each system’s whole record into every table.

What does “single reservation policy” mean?

Public create, vendor manual create, customer reschedule, and vendor reschedule already call the shared reservation/visit helpers. Future waitlist promotion and calendar sync must use the same helpers rather than implement their own availability check. Only approved staff actions may use explicit overrides, and those overrides must be logged.

Click-by-click plan

The order to tackle it, with concrete work at every step

The target is a general-purpose appointment and reservation engine, not a salon-only feature. A salon, clinic, consultant, gym, equipment rental business, or meeting-space operator should configure the same engine differently. The order below builds the shared rules first and the industry-specific screens second.

Confirmed decisions: an appointment may contain multiple services and reserve multiple internal capacity records; staff availability comes from Workforce shifts and approved leave; each workspace configures deposits, cancellation/rescheduling/no-show rules and permitted notification channels; pay-at-venue stays unpaid until staff record online or cash settlement.
Implemented in the current codebase

Public storefront integration: one branded website, independently published Ordering and Booking features

The functional customer booking flow now lives on the same branded public business site as ecommerce. Its canonical address is /storefront/:slug/Book. A business may publish Ordering only, Booking only, both, or neither.

What is now built: migration 157_booking_publication adds Booking-owned, default-off public_enabled state. The workspace Storefront screen now owns independent Online ordering and Online booking toggles. Booking Setup configures the booking service; Storefront makes it customer-visible only when setup is complete and an active service exists. Every Booking publication change is authenticated, protected against cross-site requests and added to the audit log. Migration 156 remains reserved for notification jobs.

/storefront/:slug resolves to the first enabled public feature, and the shared public navigation shows Ordering and/or Booking independently. Booking public service, availability, capacity, and create APIs return unavailable until Booking is enabled, setup is completed, a service is active, and the owner has published it. Older /book/:slug and /c/:slug/book URLs redirect into the shared storefront shell.

Remaining before release: add the owner-facing copy-link and QR-code actions, exercise Ordering-only, Booking-only, both and neither in a real browser on desktop and mobile, and confirm migration 157 is applied to the intended deployment database before promotion. Steps 9 (Payments) and 10 (notifications/integrations) are still pending.

Completed: Step 1 Rename the product concept and write the v1 contract

Implemented: the customer-facing product label is Appointments & Reservations. The historic saloon-booking registry key remains stable intentionally; changing a live key remains a separately reviewed compatibility migration, not a copy edit.

This roadmap is the current v1 contract: general-purpose appointments and reservations, one visit with sequential service lines, staff or non-staff capacity, pay-at-venue, Workspace-defined policies, Workforce-backed staff availability, and a shared public storefront. Multi-location, online settlement, notifications, and advanced capacity are intentionally deferred.

Acceptance: a developer can explain the product without using salon-specific terms, and every future data field maps to a written business rule.

Completed: Step 2 Build Booking Setup onboarding before any capacity-management screen

Implemented: Booking Setup replaces the client-facing Resources page. Its five-step flow persists editable answers for booking party, bookable kinds, extra capacity, availability source, and display labels in booking_setup. The old Resources route redirects here.

The client can reopen and edit the answers. Setup completion and an active service are prerequisites for publishing Booking. Current navigation remains fixed rather than generating tabs; this prevents a generic Resources surface while keeping the workspace predictable.

The service editor uses plain questions: “This service can be performed by…”, “This service needs…”, and “This booking uses…”. Internally these answers create generic resource records and eligibility/reservation rules. Do not expose that internal word in the UI.

Remaining acceptance: add configuration-path presentation tests and one label resolver so a solo consultant sees Your Availability, a salon sees Stylists/Stations, a clinic sees Doctors, Rooms, Equipment, and a rental business sees Assets. No client-facing configuration path may display a generic “Resources” label or page.

Completed: Step 3 Make Workforce the staff-availability authority

Implemented: when Booking Setup selects Workforce, availability joins Booking capacity to Workforce employee profiles and reads active employment, shifts, and approved leave. Workforce hours replace the legacy Booking weekly schedule for staff booking; an off-shift or terminated employee is not offered.

Booking-owned time off remains for non-staff capacity and exceptional local blocks. A manual availability source remains available for businesses that do not use Workforce. The implementation tests immediate shift changes, approved leave, inactive employees, and tenant isolation.

Guardrail: future staff-related features must read Workforce rather than creating a competing schedule in Booking.

Completed: Step 4 Build one authoritative reservation-validation service

Implemented: public create, standard vendor manual create, guest reschedule, and vendor reschedule use the server-only reservation and visit helpers. The browser requests slots, but the server validates the requested time and the database exclusion constraint is the final concurrency authority.

It validates tenant scope, service/capacity eligibility, Workforce shifts, leave, business or manual hours, buffers, lead time, timezone, and live overlap before writing. Future waitlist promotion, external calendar sync, and advanced capacity must call this same seam.

Verified acceptance: forged requests cannot reserve an unavailable time and concurrent create coverage confirms one durable winner with safe conflicts. The blocked-time compatibility path is intentionally separate and must stay an explicit staff-only override.

Completed: Step 5 Introduce a visit with appointment lines

Implemented: migration 153_booking_visits_appointment_lines.sql adds a parent visit, sequential appointment lines, protected per-line reservations, and a primary-line bridge for the existing bookings read contract. One customer can select several services, receive one confirmation, and reserve each line safely.

The delivered scope is sequential lines: a haircut followed by a beard trim is one visit with one end time and total. Parallel clinical or equipment workflows are not inferred and remain deferred until explicitly configured.

Remaining work: migrate read APIs from booking-first to visit-first only after bridge consistency is proven. Do not remove bookings before that separately reviewed migration.

Completed: Step 6 Give each workspace a configurable policy engine

Implemented: Booking Rules persists workspace cancellation, reschedule, no-show, late-arrival, settlement, deposit, and fee choices in booking_policies. The relevant policy snapshot is stored against a visit, so later workspace edits do not silently rewrite an existing agreement.

Industry-standard baseline: allow free customer cancellation/reschedule until a configurable cutoff (often 24 or 48 hours); after it, require staff approval or apply the configured deposit/fee rule. Treat late arrival as a configurable grace period, then mark no-show only through a staff action or an explicit automated rule. Staff may override any rule only with a reason recorded in history.

Remaining work: online deposit collection and automatic settlement actions await Step 9. Staff overrides should require a reason wherever the policy permits them.

Completed: Step 7 Make lifecycle, payment state and history explicit

Implemented: appointment status is separate from payment status. A pay-at-venue visit can be confirmed while payment remains unpaid. The lifecycle supports create, confirm, reschedule, cancel, check-in, complete, no-show, block, and unblock.

Migration 155_booking_lifecycle_events.sql adds the append-only booking_events ledger. Current transitions record actor, source, previous/new state, timestamp, and reason where supplied. Payment provider events, refund events, and reconciliation await Payments.

Remaining work: add optimistic concurrency and require a reason for every policy override, not merely for the transitions that already collect one.

Completed: Step 8 Build the customer and front-desk flows around the policy

Implemented: the customer books from the same branded Storefront as ordering: select one or more services, choose an allowed time and capacity preference, provide contact information, choose an available payment mode, then receive confirmation, a manage link, and an ICS calendar file. Booking and Ordering publish independently from the workspace Storefront controls.

The vendor flow provides calendar/list, manual booking, blocked time, visit detail, check-in, cash recording, completion, no-show, cancellation, and reschedule actions. Booking CSS is now isolated in booking.css and uses ExSol theme tokens.

Remaining work: complete real-browser checks for Booking-only, Ordering-only, both, and neither storefront states on desktop and mobile; add copy-link/QR actions; apply the setup-derived label resolver throughout this flow.

Pending: Step 9 Integrate the Payments module

Do later, in the Payments chat: build the provider-neutral payment rail with Razorpay test credentials held only in environment variables. It creates payment orders/intents, verifies signed webhooks against the raw body, records idempotent provider events, supports refunds, and exposes a verified payment result to Booking.

Booking will call the Payments API with a visit ID and amount due. It never stores a Razorpay secret. Browser redirect success is only a hint; the signed webhook and payment record are authoritative. For cash, a permitted staff member records the amount/tender/reference and the event ledger records who confirmed it.

Acceptance: duplicate webhook deliveries have no double effect; online and cash payments both settle the same booking balance; a failed/expired deposit releases the hold according to policy.

Pending: Step 10 Deliver notifications, integrations, reporting and advanced features

Do: turn booking events into durable notification jobs for email, SMS, WhatsApp and push only after each channel is configured and consented. Then connect CRM customer timelines, Workforce utilization, POS settlement, Finance recognition, and Analytics read models using visit and event IDs.

Add privacy operations alongside the integrations: retention/anonymisation jobs, consent history, export/correction/deletion requests, and access audits. Measure conversion, resource utilization, lead time, cancellation and no-show rate, payment collection and service performance. Only after these are correct should the module add waitlists, recurring visits, packages, memberships, coupons, intake forms, multi-location, external calendars and capacity/group bookings.

Acceptance: every downstream report can trace its number back to an immutable booking/payment event, and every advanced feature reuses the same reservation service.

Product decisions

Adopted decisions and planned safeguards

The following product decisions guide the implementation. “Implemented” identifies a rule already enforced by current code. “Planned safeguard” identifies an accepted target that must not be represented as live behavior until its data model, UI, and tests exist.

  • Product scope (implemented): a configurable general-purpose appointment and reservation system, not a salon product. The historic saloon-booking registry key is retained for compatibility.
  • Internal capacity term (implemented): resource means scarce capacity in database/API code: staff, space, equipment, or another reservable asset. Clients never need that term; staff availability derives from Workforce shifts and approved leave when Workforce is selected.
  • One visit, many services (implemented sequentially): the customer gets one booking/confirmation for several sequential appointment lines. Parallel lines and multi-capacity orchestration are planned only after explicit business configuration.
  • Customer policy (implemented v1): each workspace configures cancellation, rescheduling, late-arrival and no-show rules, deposits/fees/refunds/credits, and staff override permission. Industry-standard default behavior is a configurable free-change window, followed by a staff approval or configured charge/deposit rule.
  • Payments (partial): payment choice and pay-at-venue state are client-configurable. A pay-at-venue visit stays open until staff records cash or a future Payments-module integration verifies online settlement. Online payments remain disabled until Step 9 is complete.
  • Channels (planned): email, SMS, WhatsApp and push are all potential channels; each requires provider setup, customer consent rules and delivery monitoring.
  • Workforce integration: staff availability must follow actual Workforce shifts and leave, so an off-shift doctor cannot be offered as bookable.
  • POS/Payments interaction (target contract): front desk retains the payment option for pay-at-venue; a shared payment record will become confirmed through the Payments module or an authorized cash-received action.
Planned guest-manage security safeguard, not implemented. The current private manage page relies on an unguessable manage token. Before high-assurance customer self-service is released, destructive actions must require a one-time code sent to the booking’s verified email or phone. The code should expire after 10 minutes, be single-use and rate-limited, and affected tokens/sessions should be revoked after cancellation or a material change. A logged-in customer should use their normal account instead.
Planned global privacy baseline, not implemented. Booking must collect only information necessary to provide the reservation and transaction; transactional booking messages and marketing consent are separate; sensitive medical/clinical notes are out of the generic Booking core until a sector-specific privacy design exists. The platform must provide role-based access, audit history, tenant isolation, customer data export/correction/ deletion workflows, a documented retention schedule, and purge/anonymisation jobs.

A global product still needs a jurisdiction review before launching in each market. The baseline follows widely used privacy principles: purpose limitation, data minimisation, security, accountability and keeping identifiable data no longer than justified. The system must record which policy/version applies to a customer and avoid treating consent for an appointment as consent for marketing.

Planned external-failure defaults, not implemented. A payment-required visit should hold capacity for 15 minutes, then expire according to the workspace policy. Payment provider events must be stored idempotently; a late event must never silently reopen a taken slot and instead enter a staff-visible reconciliation queue. Notification delivery should retry with backoff for up to 24 hours; permanent failure should create a visible staff action without cancelling the booking. Calendar-sync failure must leave the ExSol booking unchanged, retry safely, and show its sync state to staff.

These targets belong in the technical design, migrations, and tests for Steps 9 and 10. The detailed legal/privacy implementation still requires review for each target jurisdiction and industry.

Release bar

What remains before we call Booking production-ready?

Correctness

  • Implemented: public create, standard manual create, and guest/vendor reschedule use shared server-side reservation validation. Future writers must use it too.
  • Implemented: database constraints prevent double-booking under concurrent requests.
  • Implemented: timezone/DST, buffers, lead time, availability and key cutoffs are tested.
  • Partial: lifecycle changes are authorized and evented; optimistic concurrency and mandatory override reasons remain.

Operations

  • Implemented v1: staff can configure services and Booking Setup, manage a day, create bookings or blocks, cancel/reschedule, check in, record cash, and close appointments.
  • Missing: notifications must be queued, observable and retryable.
  • Partial: surface errors now show retry actions; reconciliation and recovery work remains.

Platform quality

  • Partial: module CSS is isolated and scheduling writes are shared; shared permissions and setup-derived presentation labels still need extraction.
  • Implemented: auth gates include product enablement and Owner bypass.
  • Full npm run typecheck and npm test are green, and all public storefront states are verified against a real deployment on desktop and mobile.

Money and data

  • Required: online payment remains disabled until verified webhook, idempotency and reconciliation are live.
  • Partial: Finance/Analytics consume booking outcomes; documented payment and visit-first read contracts remain.
  • Required: customer identity and consent ownership must be clear across Booking, CRM and Marketing.
