Read this first

The honest short answer

ExSol Booking is a functional v1 appointment scheduler, not a finished booking product. It can configure services, staff or rooms, hours and time off; calculate open slots; accept a guest booking; prevent two live bookings from occupying the same resource at the same time; let staff view and operate bookings; and let a guest cancel through a private link.

Safe present-day promise: a small single-location service business can demonstrate pay-at-venue appointment booking with manually managed availability. Unsafe promise: production-grade online payments, dependable guest rescheduling, notification delivery, reporting, auditability, multi-location operations, or a unified customer record.

This document calls a feature good when its core rule is enforced on the server and has meaningful tests, repair when the UI or API can violate its intended business rule, and missing when a normal production operator would expect it but the system has no complete implementation.

First principles

What a booking module really does

A booking module is a capacity-allocation system. A customer asks for a service at a time. The system must assign a resource that can perform it, reserve that capacity once, communicate the outcome, and preserve a trustworthy history for operations and finance.

Input

Demand

Customer, selected service, desired time, staff preference, contact consent, payment choice.

Decision

Availability

Business hours, resource hours, holidays, leave, service duration, buffers, lead time, existing reservations.

Outcome

Reservation

One durable appointment record, a lifecycle state, a payment state, notifications, and an audit trail.

Important distinction: a slot is usually not stored as a database row. In ExSol it is calculated when requested. The durable thing is the appointment time range. That is a valid and scalable design when the database protects against overlaps.
Client-facing language rule: customers and workspace operators never see the generic word “resource.” It remains an internal database/API term meaning scarce capacity. The client configures real-world things such as team members, doctors, rooms, stations, vehicles, equipment, or availability through Booking Setup.
What customers see instead: Booking Setup

The first visit to Booking Setup launches guided onboarding. It asks plain-language business questions, creates the internal capacity records and reservation rules, then hides technical terminology permanently.

Who do customers book with?
Specific team member · Any available team member · Nobody specific, just a time/place
What can be booked?
Appointments · Rooms/spaces · Equipment/assets · A combination
Does each booking need anything besides a team member?
No · A room/space · Equipment · Both
Where does availability come from?
Workforce shifts and leave · Manual business hours for now
Connect services in plain language.
“This service can be performed by…” · “This service needs…” · “This booking uses…”

After setup, the navigation is contextual: Team Availability, Rooms & Spaces, Equipment, and Booking Rules appear only when relevant. A solo consultant sees Your Availability; a salon sees Team Availability and optional Stations; a clinic sees Doctors, Rooms, and Equipment; an equipment-rental business sees Assets and Booking Rules.

Engineering assessment

What is good, bad, and missing

Good

The database owns “never double book”

The EXCLUDE USING gist constraint is the right authority. Two concurrent requests cannot both create overlapping live appointments for one resource. This is materially better than trusting a browser’s displayed slot list.

Good

Scheduling logic is pure and tested

Time-zone conversion, schedule intersection, buffers, and least-busy assignment live in small reusable functions. This makes the highest-risk calculations testable without HTTP or a browser.

Good

Tenant isolation is deliberate

Vendor queries scope records by bucket_id/clientId; private guest management uses an unguessable token; settings and resources belong to one client.

Good

Access fits the platform

The module is registered correctly, enabled through a product, has dedicated navigation, uses platform permission keys, and includes L1 Owner bypass in server and client gates.

Repair first

Rescheduling bypasses business rules

The guest manage endpoint directly changes time_range. It does not verify that the new time is an advertised slot, respects resource working hours/time off, matches service eligibility, or observes lead time. The overlap constraint catches only collisions. This must be routed through one shared reservation validator.

Repair first

Public creation trusts too much

A named resource booking verifies that the resource exists, but does not verify that it is eligible for the chosen service or that the chosen start is inside working hours and outside time off. The UI normally supplies valid data; the server must independently enforce it.

Repair

Payment is a false-complete surface

Paid service modes have a database enum and a stub response, not Razorpay order creation, checkout, verified state reconciliation, idempotency, refund flow, or payout reconciliation. Keeping the UI gate off is correct; do not turn it on early.

Repair

Module boundaries are weak in the frontend

Booking styles sit in the large shared src/lib/components.css, contrary to the repository’s namespaced per-module CSS rule. This increases coupling and makes visual regressions harder to isolate.

Missing

Notification reliability

No complete reminder, cancellation, reschedule, staff alert, retry, delivery-status, SMS/WhatsApp, or customer preference system exists. A booking product without reminders will generate avoidable no-shows.

Missing

Operational audit and recovery

There is no appointment event history, actor record for each change, optimistic versioning, admin override reason, or payment-to-booking reconciliation view. Operators cannot explain “who changed this and why?”

Missing

Business breadth

No multi-location capacity model, resource skills, recurring appointments, waitlist, packages/memberships, coupons, intake forms, deposits/refunds, external calendars, or customer self-service authentication.

Missing

Measurement

Analytics and Finance consume simple booking outcomes/revenue, but Booking lacks its own dashboard for occupancy, cancellation/no-show rate, lead time, resource utilization, conversion, and cohort retention.

Priority rule: do not add attractive features such as memberships or calendar sync before server-side booking validation is centralized. A reliable scheduling kernel is the prerequisite for every later capability.
Connection map

How the codebase connects a click to a database row

The browser never talks directly to Postgres. It renders React components, calls a same-origin API route, the Netlify Function authenticates and validates, then it reads or writes Neon Postgres. This separation prevents users from bypassing the interface. 

Guest clicks “Confirm booking”
  BookingStorefront / Checkout.tsx
      ↓ bookingPublicApi.create(...)
  POST /api/booking-public/:slug/create
      ↓ booking-public-create.ts
  Zod validation → rate limit → tenant/service/resource checks → customer upsert
      ↓
  INSERT public.bookings (... time_range ...)
      ↓ PostgreSQL exclusion constraint either accepts or rejects overlap
  JSON response → confirmation screen / manage link / ICS file
Route: src/lib/router.tsx maps public /c/:slug/book and vendor /c/:slug/booking/* URLs to components.
UI: public and vendor React components call typed wrapper functions in src/modules/booking/shared/api.ts.
Function: each flat netlify/functions/booking-*.ts file declares its HTTP route and method, parses input, authorizes, and returns JSON.
Domain logic: pure scheduling and state-transition rules live in src/modules/booking/lib/. These must become the only source for all validation paths.
Storage: database tables record the durable truth; the query result becomes API JSON; React stores only temporary display state.
Why both client and server permissions?

The client gate keeps a user from seeing a page they cannot use. The server gate is the security boundary: a user can always forge an HTTP request from developer tools, so every mutation must verify session, product enablement, owner bypass, and permission on the server. ExSol’s requireBooking wrapper is built for this.

Why store money in cents and time in UTC?

Decimal floating-point arithmetic can produce errors such as 0.1 + 0.2. Integer cents avoid that. Local times are ambiguous around daylight-saving changes; UTC instants are not. Store tstzrange in UTC, but calculate and display business hours in the tenant’s IANA timezone such as Asia/Kolkata.

What should never be the source of truth?

A React state variable, browser local storage, a cached availability response, an email, or a payment-provider redirect must not be the appointment truth. They are conveniences. PostgreSQL appointment records plus a verified payment/event ledger are the truth.

Platform map

Which ExSol modules Booking needs, and who needs Booking

Booking consumes

Dependencies

  • Platform identity and access: tenant, user session, permission matrix, user nodes.
  • Payments: provider abstraction, order/payment/refund records, signed webhooks.
  • Email: transactional templates and delivery outbox.
  • Workforce: staff identities, shifts, leave, timesheets; currently it already reads booking resources.
  • CRM: canonical customer profile, notes, consent, timeline.
  • Branding: public storefront identity, logo, colors, domains.
  • Files: intake attachments, consent forms, before/after media, only if business needs it.
Booking produces

Downstream data

  • Analytics: booking outcomes are already queried by analytics-bookings.ts.
  • Finance: confirmed/completed booking price is already included as a revenue channel; payment settlement must eventually refine this.
  • Marketing: current ROI links completed bookings by email; future segmentation needs canonical customer IDs and consent.
  • CRM: appointment timeline, visit history, no-show/cancellation facts, preferences.
  • Workforce: utilization, commissions, attendance versus scheduled work.
  • POS/Orders: check-in and pay-at-venue collection should refer to the same appointment/payment record, not duplicate a sale.
Ownership rule: Booking owns appointment facts. Payments owns provider transactions. CRM owns rich customer profile. Workforce owns employment and shifts. Finance owns accounting treatment. Integration means reference IDs and events, not copying each system’s whole record into every table.

What does “single reservation policy” mean?

Public create, vendor manual create, customer reschedule, vendor reschedule, waitlist promotion, and calendar sync must not each implement a slightly different availability check. They should call one service that receives a request, explicit override flags, and a current database view. Only approved staff actions may use explicit overrides, and those overrides must be logged.

Click-by-click plan

The order to tackle it, with concrete work at every step

The target is a general-purpose appointment and reservation engine, not a salon-only feature. A salon, clinic, consultant, gym, equipment rental business, or meeting-space operator should configure the same engine differently. The order below builds the shared rules first and the industry-specific screens second.

Confirmed decisions: an appointment may contain multiple services and reserve multiple resources; staff availability comes from Workforce shifts and approved leave; each workspace configures deposits, cancellation/rescheduling/no-show rules and permitted notification channels; pay-at-venue stays unpaid until staff record online or cash settlement.

Rename the product concept and write the v1 contract

Do: retire the misleading product label “Saloon Booking System” in favor of a neutral customer-facing name such as “Appointments & Reservations.” Keep a compatibility decision for the existing registry/product key: changing a live key is a migration project, not a text edit.

Write a one-page v1 contract: supported appointment types, one or many locations, currencies, staff versus non-staff resources, multi-service visit behavior, customer/employee roles, payment states, notification channels, global privacy baseline, and explicit items deferred to v2. Do this before any migration is proposed.

Acceptance: a developer can explain the product without using salon-specific terms, and every future data field maps to a written business rule.

Build Booking Setup onboarding before any capacity-management screen

Do: replace the generic client-facing “Resources” page with Booking Setup. On the first visit, save each answer as editable onboarding state: who customers book with (specific team member, any available team member, or nobody specific), what can be booked (appointments, rooms/spaces, equipment/assets, or a combination), whether a booking needs a team member plus a room/space and/or equipment, and whether availability comes from Workforce or manual business hours for now.

Save a booking_setup record with the answers, setup version/completion state, enabled internal types, and display labels. Generate safe defaults from the answers: navigation sections, default reservation rules, service wording, and the availability-source choice. The client can reopen and edit Booking Setup later; edits must show what will change before they affect new bookings.

The service editor uses plain questions: “This service can be performed by…”, “This service needs…”, and “This booking uses…”. Internally these answers create generic resource records and eligibility/reservation rules. Do not expose that internal word in the UI.

Acceptance: tests cover every onboarding branch and verify the resulting UI: solo consultant → Your Availability; salon → Team Availability and optional Stations; clinic → Doctors, Rooms, Equipment; rental business → Assets and Booking Rules. No configuration path displays a generic “Resources” label or page.

Make Workforce the staff-availability authority

Do: define the contract between Workforce and Booking. Workforce owns employee identity, published shifts, approved leave, and employment status. When Booking Setup selects specific or any team member, Booking reads that availability and must not create a second competing staff schedule. Internal time-off records remain only for non-staff capacity or exceptional local blocks.

Add a resource-to-employee link, query shift/leave intervals in availability, and decide the fallback for a resource not linked to a worker. If an employee’s 3pm shift starts at 7pm, a 3pm slot must never be returned, even if the booking resource’s old weekly JSON says it is open.

Acceptance: create a Workforce shift, request slots, change or cancel the shift, and prove the slot list changes immediately; test leave, inactive employees, and cross-tenant isolation.

Build one authoritative reservation-validation service

Do: trace every appointment-changing endpoint: public create, vendor manual create, customer reschedule, vendor reschedule, waitlist promotion and future calendar sync. Make all of them call one server-only reservation service. The browser may ask for availability, but the server makes the final decision.

The service must validate tenant/location, service/resource eligibility, Workforce shifts, resource hours, closures, leave, buffers, lead time, cancellation cutoff, appointment policy, timezone, and live overlap. It then performs the insert/update within the database protection already provided by the exclusion constraint.

Acceptance: a forged API request cannot reserve an unavailable time; public create and reschedule have identical rule coverage; ten simultaneous requests yield one winner and nine safe conflicts.

Introduce a visit with appointment lines

Do: replace the one-row-equals-one-service limitation with a parent booking_visit and child booking_appointment_lines. The visit is what the customer sees and pays for. Each line records service, duration, price snapshot, sequence, assigned resource(s), and its own time range.

First build the simple sequential case: haircut 30 minutes followed by beard trim 20 minutes, with one customer confirmation and one total. Then support parallel lines only where a business explicitly configures them, such as a doctor consultation and a machine test. Do not model complex clinical workflows before the sequential visit works.

Acceptance: a customer selects several services once, sees one visit total and one end time, and the database prevents overlap on every resource used by every line.

Give each workspace a configurable policy engine

Do: provide a Booking Policy settings screen, not hardcoded behavior. Each workspace chooses a deposit rule, cancellation cutoff, reschedule cutoff, permitted reschedule count, late-arrival grace period, no-show outcome, refund/credit rule, and whether an extra fee applies to a late reschedule.

Industry-standard baseline: allow free customer cancellation/reschedule until a configurable cutoff (often 24 or 48 hours); after it, require staff approval or apply the configured deposit/fee rule. Treat late arrival as a configurable grace period, then mark no-show only through a staff action or an explicit automated rule. Staff may override any rule only with a reason recorded in history.

Acceptance: changing a workspace policy changes what the customer can do, but never silently changes the policy snapshot already agreed for an existing visit.

Make lifecycle, payment state and history explicit

Do: separate appointment status from payment status. Appointment status answers “is the customer expected, checked in, completed, cancelled or no-show?” Payment status answers “unpaid, payment requested, partly paid, paid, refunded, waived, or cash pending?” A pay-at-venue visit may be confirmed while payment remains unpaid.

Add an append-only booking_events ledger. Write one event for create, quote, hold expiry, confirm, reschedule, cancel, check-in, completion, no-show, payment request, payment confirmation, offline cash receipt, refund and override. Include actor, source, previous/new value, timestamp and reason.

Acceptance: a staff member can open a visit and answer “what happened, who did it, what is owed, and why?” without reading database logs.

Build the customer and front-desk flows around the policy

Do: revise the storefront flow to become: choose location (when enabled) → choose one or more services → choose staff preference if permitted → choose a valid visit time → see itemized price/deposit/policy → supply contact and consent → select payment method → receive confirmation. The manage flow must show only actions allowed by that visit’s policy snapshot.

Build the vendor flow in parallel: calendar and list → open visit → check in → record cash/offline payment or send online payment link → complete/no-show/reschedule/cancel with reason. Move global Booking CSS into booking.css and use only ExSol theme tokens.

Acceptance: a customer can move to an open allowed slot and their original capacity is released atomically; a receptionist can complete the same process on behalf of a caller.

Integrate the Payments module after the Booking contract is stable

Do later, in the Payments chat: build the provider-neutral payment rail with Razorpay test credentials held only in environment variables. It creates payment orders/intents, verifies signed webhooks against the raw body, records idempotent provider events, supports refunds, and exposes a verified payment result to Booking.

Booking will call the Payments API with a visit ID and amount due. It never stores a Razorpay secret. Browser redirect success is only a hint; the signed webhook and payment record are authoritative. For cash, a permitted staff member records the amount/tender/reference and the event ledger records who confirmed it.

Acceptance: duplicate webhook deliveries have no double effect; online and cash payments both settle the same booking balance; a failed/expired deposit releases the hold according to policy.

Deliver notifications, integrations, reporting and advanced features

Do: turn booking events into durable notification jobs for email, SMS, WhatsApp and push only after each channel is configured and consented. Then connect CRM customer timelines, Workforce utilization, POS settlement, Finance recognition, and Analytics read models using visit and event IDs.

Add privacy operations alongside the integrations: retention/anonymisation jobs, consent history, export/correction/deletion requests, and access audits. Measure conversion, resource utilization, lead time, cancellation and no-show rate, payment collection and service performance. Only after these are correct should the module add waitlists, recurring visits, packages, memberships, coupons, intake forms, multi-location, external calendars and capacity/group bookings.

Acceptance: every downstream report can trace its number back to an immutable booking/payment event, and every advanced feature reuses the same reservation service.

Product decisions

Adopted product and operating policies

The following decisions are now part of the roadmap. They are not implementation details: each changes the data model, permissions, policy, screens, integrations and operational obligations.

  • Product scope: a configurable general-purpose appointment and reservation system, not a salon product. The historic saloon-booking registry key requires a separate compatibility/rename decision.
  • Resource: any scarce capacity. It can be staff, space, equipment or another reservable asset. Staff resources must derive availability from Workforce shifts and leave.
  • One visit, many services: the customer gets one booking/confirmation for several sequential or explicitly configured parallel appointment lines. The model must support multiple resources per visit.
  • Customer policy: each workspace configures cancellation, rescheduling, late-arrival and no-show rules, deposits/fees/refunds/credits, and staff override permission. Industry-standard default behavior is a configurable free-change window, followed by a staff approval or configured charge/deposit rule.
  • Payments: deposits and payment choice are client-configurable. A pay-at-venue visit stays open until verified online payment or staff-recorded offline cash payment settles it.
  • Channels: email, SMS, WhatsApp and push are all potential channels; each requires provider setup, customer consent rules and delivery monitoring.
  • Workforce integration: staff availability must follow actual Workforce shifts and leave, so an off-shift doctor cannot be offered as bookable.
  • POS/Payments interaction: front desk retains the payment option for pay-at-venue; a shared payment record becomes confirmed through the Payments module or an authorized cash-received action.
Adopted guest-manage security default. The private manage URL starts a session but is not sufficient for a destructive action. Viewing is limited to the booking summary; cancel, reschedule, contact-detail change and payment-link actions require a one-time code sent to the booking’s verified email or phone. The code expires after 10 minutes, can be used once, is rate-limited, and every existing manage session is revoked when the booking is cancelled or materially changed. A logged-in customer uses their normal account instead.
Adopted global privacy baseline. Booking collects only information necessary to provide the reservation and transaction; transactional booking messages and marketing consent are separate; sensitive medical/clinical notes are out of the generic Booking core until a sector-specific privacy design exists. The platform must provide role-based access, audit history, tenant isolation, customer data export/correction/ deletion workflows, a documented retention schedule, and purge/anonymisation jobs.

A global product still needs a jurisdiction review before launching in each market. The baseline follows widely used privacy principles: purpose limitation, data minimisation, security, accountability and keeping identifiable data no longer than justified. The system must record which policy/version applies to a customer and avoid treating consent for an appointment as consent for marketing.

Adopted external-failure defaults. A payment-required visit holds the resource for 15 minutes, then expires according to the workspace policy. Payment provider events are stored idempotently; a late event never silently reopens a now-taken slot and instead enters a staff-visible reconciliation queue. Notification delivery is retried with backoff for up to 24 hours; permanent failure creates a visible action for staff but does not cancel the booking. Calendar-sync failure leaves the ExSol booking unchanged, retries safely, and shows its sync state to staff.

These defaults now belong in the technical design, migrations and tests. They are product commitments; the detailed legal/privacy implementation still requires review for each target jurisdiction and industry.

Release bar

When can we call Booking production-ready?

Correctness

  • Every create and reschedule path uses one server-side reservation validation service.
  • Database prevents double-booking under concurrent requests.
  • Timezone/DST, buffers, closures, lead time and cutoffs are tested.
  • Lifecycle changes are valid, authorized and auditable.

Operations

  • Staff can configure catalogue/capacity, manage a full day, search, cancel/reschedule, check in and close appointments.
  • Notifications are queued, observable and retryable.
  • Failures have clear user messages and an operator recovery path.

Platform quality

  • Module has isolated CSS, typed API contracts and no duplicated scheduling logic.
  • All auth gates include product enablement and Owner bypass.
  • Full npm run typecheck and npm test are green; browser flows are verified against a real deployment.

Money and data

  • Online payment remains disabled until verified webhook, idempotency and reconciliation are live.
  • Finance/Analytics consume explicit, documented booking events.
  • Customer identity and consent ownership are clear across Booking, CRM and Marketing.
Recommended next engineering task: write a narrow technical design for the shared reservation-validation service and its test matrix, then repair public create and guest reschedule to use it. This is the smallest high-leverage change because it turns the existing strong exclusion constraint into a complete scheduling guarantee.
