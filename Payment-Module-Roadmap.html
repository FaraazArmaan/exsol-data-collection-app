Read this first

The honest short answer

ExSol does not have a working Payments module today. It has a Payments label in the module registry, a product entry and valid permission rows. Around that empty shell, Booking and POS each have payment-adjacent fields, status changes and partial workflows. Those are useful integration seams, but they are not a ledger and must not be mistaken for a complete money system.

Safe present-day promise: staff can record cash-like payment state in Booking and POS can mark a sale paid under its existing permissioned state machine. Not ready to promise: online collection, a provider order, verified settlement, refunds through a provider, payment receipts, duplicate-webhook safety across the real Booking model, tenant payment settings, or reliable payment reporting.

A safe Payments module should own the question “what money was requested, attempted, captured, refunded, or still owed?” Booking should continue to own appointments; POS should continue to own sales and fulfilment. Finance and Analytics should read payment evidence rather than infer money from an appointment or sale status.

Recommended first milestone: build the shared ledger and staff cash receipts for Booking before touching online money. That proves the data model, tenant scope, permissions and UI. Then build one Razorpay Booking deposit flow end to end, harden it against retries, and reuse the same core for POS/storefront.
Update: Razorpay Test keys are available. This unblocks sandbox order creation and Checkout testing; it does not make production payments ready. Keep the downloaded key pair out of Git, source files, browser bundles, screenshots, test fixtures and this roadmap. The Key ID may be returned only as part of a specific Checkout response; the Key Secret stays server-only. Before webhook testing, create a separate Test-mode webhook secret in Razorpay Dashboard and use a publicly reachable staging URL—Razorpay cannot deliver webhooks directly to localhost. Test and Live mode connections, credentials and webhook URLs remain separate.
First principles

What a payment module really does

A payment module is not just “a Pay now button.” It is the system of record for a financial promise between a business and a customer. It must keep the business state and the money state separate, because money can arrive late, fail, be partly paid, be refunded, or be disputed.

Input

Obligation

Booking or POS says what is owed: for example, “₹500 appointment deposit” or “₹1,249 order total.” The source gives a stable ID and a server-calculated amount snapshot.

Decision

Collection

Payments creates a request and an attempt: a staff cash receipt, a Razorpay order, or later a payment link. It never trusts the browser to choose the final price.

Outcome

Evidence

A verified capture or cash receipt becomes an immutable transaction. An allocation says which obligation it settled. A refund becomes a separate negative transaction.

The essential separation: a Booking visit can be confirmed while its payment is cash pending. A POS sale can be paid while fulfilment is still pending. “Appointment/order status” and “payment status” must never be one overloaded field.

Engineering assessment

What is good, bad, and missing

Good

The source modules already protect prices

POS and public storefront calculate price, tax and discounts on the server before persisting a sale. Payments should consume that snapshot—not ask a React page to send an amount it chose.

Good

Payment and fulfilment are partly separated

POS distinguishes pending payment, paid and fulfilled. Booking distinguishes appointment state from payment state. This is the correct conceptual direction.

Good

Security primitives already exist

The shared webhook helper documents raw-body HMAC verification and timing-safe comparison. Sessions, tenant IDs, audit logging and module authz factories already exist.

Good

Money is already integer minor units

Amounts are stored in integer *_cents fields. In an INR-first product these represent paise; the new module should use explicit amount_minor plus currency.

Repair first

There is no authoritative payment record

Several modules write their own payment-like fields. No row answers: which provider transaction settled which booking/sale, who recorded it, what amount remains, and whether a refund has happened.

Repair first

Provider identity is mixed with idempotency

sales.payment_ref contains idem:<key> for create replay. Reusing it for Razorpay would make retry safety and provider reconciliation fight each other.

Repair

Booking online payment is false-complete

The UI feature flag is correctly false. The stub intent and legacy webhook test can make a reader think Razorpay works, but no safe end-to-end online journey exists.

Repair

Reporting is business-state based

Finance counts confirmed/completed Booking rows as revenue and paid/fulfilled sales as income. That is useful operational reporting, but it is not a complete cash collection or refund ledger.

Missing

Provider connection management

Per-tenant credentials, write-only settings, encrypted secret storage, connection health, environment key rotation and audit records do not exist.

Missing

Webhooks as an inbox

There is no durable event table, provider event uniqueness constraint, processing status, retry path, late-event policy or staff reconciliation queue.

Missing

Partial payment and refunds

One transaction needs to allocate to an obligation; a negative refund must point back to the original transaction. Neither is a cross-module primitive today.

Missing

Operations and support

Staff need a payment timeline, receipt references, exceptions, stale checkout expiry, “customer paid but order pending” recovery and provider outage guidance.

Non-negotiable rule: the browser’s Razorpay “payment successful” callback is advisory only. It may navigate a customer to a receipt screen, but it must never be accepted as proof of money. Only a verified provider event or an authorized staff-recorded offline receipt can create a successful transaction.
Connection map

How the codebase should connect a click to a money record

The pattern is deliberately boring: a screen calls a typed API wrapper; a function validates/authenticates; a server-only Payments service writes the ledger; then a narrow outcome is returned to the source module. Booking and POS never import each other’s internals, and a payment provider never talks directly to the browser as the authority.

Booking or POS
Owns the appointment or sale and sends a tenant-scoped source ID plus its server-authoritative amount snapshot.
Payments API
Creates a payment request and, for online flow, an attempt with a dedicated provider order ID.
Hosted provider Checkout
Customer enters card/UPI data at Razorpay. ExSol handles no card data and only displays pending/verified state.
Verified webhook inbox
Raw body → resolve tenant connection → HMAC signature → unique provider event → amount/currency check.
Transaction and allocation
Write immutable capture/refund/cash transaction and apply its amount to the booking or sale request.
Source outcome and readers
Booking may confirm a held visit; POS may transition paid. Finance, Analytics and Marketing read documented outcomes.
What happens when a receptionist clicks “Record cash”?

The booking page calls paymentsApi.recordCash(...), not a custom inline fetch. The handler validates the request and payments.customers.create permission, checks the visit belongs to the staff member’s tenant, calculates outstanding balance from allocations, inserts one immutable cash_received transaction and allocation in a database transaction, records audit evidence, and returns the new payment summary. Booking uses that result to update its own payment display and only changes appointment state if its policy explicitly says that a satisfied deposit confirms the visit.

Storage design

What is stored, and what should be stored next

Postgres is the system of record. Use Netlify Blobs only for an optional receipt file later; do not put payment state, secrets, card data or raw credentials in Blobs. The first migration needs a human-coordinator-assigned number. The historical “051 Payments” reservation is not available now: later migrations already run through 157.

Storage rules that make the ledger trustworthy
  • Store integer minor units and ISO currency. Never use JavaScript decimal/float math for money.
  • Never delete a transaction. Correct it with a compensating adjustment, void state or refund.
  • Provider event ID is unique; provider transaction ID is unique per provider. Both make duplicate delivery/retry harmless.
  • Keep the source snapshot because a product/service price may change after checkout was created.
  • Use one database transaction for event processing, transaction creation, allocation and the source outcome write.
  • Encrypt provider secrets using AES-256-GCM with an environment-held key-encryption key. Secrets are write-only in the UI and never logged.
Why not put payment columns directly on sales and booking_visits?

Those tables may still hold fast summary fields for display, but they cannot represent a retry, a deposit plus a cash balance, a provider capture plus a partial refund, an event that arrived twice, or an amount linked to several obligations. A separate ledger is not abstraction for its own sake; it is the minimum shape that preserves evidence and supports normal payment behaviour.

Platform map

Which ExSol modules Payments needs, and who needs Payments

Payments consumes

Dependencies

  • Booking: visit ID, payment policy, server amount, hold expiry and customer snapshot.
  • POS / Storefront: sale ID, authoritative total, channel, price/tax snapshot, customer snapshot.
  • Orders: approved refund request; it keeps customer-service workflow ownership.
  • Platform auth: tenant session, product enablement, L1 Owner bypass, permission matrix and audit helper.
  • Email: safe notification request after verified outcomes; Email owns delivery/retries.
  • Client settings: tenant identity and base currency; provider connection is Payments-owned.
Payments produces

Downstream data

  • Booking: paid/outstanding/refunded summary and receipt/reference; Booking decides its own confirmation/cancellation rule.
  • POS: payment result; POS decides paid/fulfilled state and inventory timing.
  • Orders: refund provider state and immutable refund transaction link.
  • Finance: gross captured, refunds, net collected, outstanding and later settlement data.
  • Analytics / Marketing: documented paid conversion/refund metrics, never payment credentials or sensitive instrument data.
  • CRM: safe customer timeline events, not secret payment payloads.
Boundary rule: modules never import another module’s internal files. Browser types can live in src/lib/payments-contract.ts; server orchestration lives in netlify/functions/_shared/payments-service.ts. Booking and POS call that stable seam rather than copying payment logic.

What does “provider-neutral” actually mean?

It does not mean invent a huge universal gateway framework. It means business logic calls a small internal shape such as “create order,” “verify webhook” and “request refund.” _shared/razorpay-payments.ts knows Razorpay field names; payments-service.ts knows requests, transactions and allocations. Adding Stripe later should require a second adapter, not a rewrite of Booking, POS, the database or the dashboard.

Click-by-click plan

The order to tackle it, with concrete work at every step

The target is a tenant-owned, provider-neutral money record, not a Razorpay button bolted onto Booking. Each phase gives a demonstrable safety property before the next one adds more moving parts.

Recommended operating model: each client connects its own Razorpay account, and funds settle directly to that client. ExSol does not custody money or collect marketplace commission in v1. This is much simpler operationally and avoids turning the platform into a marketplace/payout product.

Write the Payment v1 contract and freeze unsafe paths

Do: keep ONLINE_PAYMENTS_ENABLED = false and do not configure a live provider webhook to the existing Booking handler. Write the definitions for request, attempt, transaction, allocation, payment status, refund, expiry, late event, and manual cash receipt.

Acceptance: every team member agrees that current Booking/POS fields are source summaries, not the ledger; the human coordinator assigns a new migration number before schema work starts.

Build the canonical module shell and permissions

Do: create the Payments folder, shared layer, thin authz wrapper, route mount, dedicated navigation and empty dashboard/settings screens. Map existing bucket×verb permissions to practical actions; do not add action-shaped keys.

Acceptance: an enabled Owner can visit a real Payments screen; disabled product users redirect safely; non-Owner access is exactly the permission matrix; generic module rail duplicate disappears.

Build the internal ledger with Booking cash receipts first

Do: add requests, transactions and allocations. Implement a staff “record cash” flow against one booking visit, including reference, actor, partial payment, outstanding balance and overpayment prevention.

Acceptance: a ₹500 cash receipt creates immutable evidence and allocation; retry cannot create a second receipt; Booking shows a derived payment summary without becoming the authoritative ledger.

Add secure tenant provider configuration

Do: use the available Razorpay Test-mode Key ID/Key Secret only in local or staging server configuration while this phase is built. Store the Key ID plus encrypted API/webhook secrets per tenant. Make settings write-only, audit changed field names not values, and reject online enablement until all required credentials are present. Create a distinct Test-mode webhook secret in the Razorpay Dashboard; it is not the API Key Secret.

Acceptance: secrets are never returned or logged; missing/wrong encryption key fails closed; different tenants cannot read or use each other’s connection; Test credentials cannot accidentally enable a Live-mode collection path.

Implement one online Booking deposit flow end to end

Do: create a request from a pending Booking visit, create one Razorpay order/attempt, launch hosted Checkout and process the raw signed webhook into event → capture transaction → allocation → Booking outcome.

Acceptance: duplicate webhook deliveries create one capture; wrong amount/currency is quarantined; browser success does not confirm the visit; an expired hold with late payment is visible to staff rather than silently reopening capacity.

Reuse the core for POS and public storefront sales

Do: after existing sale code has calculated price/coupon/tax server-side, create a Payments request for the sale. Keep POS sale FSM and inventory timing as their own domain rules; do not put provider IDs in payment_ref.

Acceptance: an online sale has one provider order, one captured transaction and one allowed pending_payment → paid result; fulfilment remains a separate staff event.

Connect Orders refunds to actual money movement

Do: retain the Orders approval workflow, but have an approved request call Payments. Payments validates remaining refundable amount, creates a pending refund transaction and waits for provider confirmation.

Acceptance: partial refunds are supported; duplicate requests cannot over-refund; a sale/booking status does not pretend to be refunded until the payment outcome is known.

Build reconciliation, reporting and operations

Do: add payment timeline, exception queue, stale attempt expiry, webhook reprocess path, alerting and staff runbook. Define Finance views for captured/refunded/net/outstanding and decide separately whether P&L uses cash or accrual recognition.

Acceptance: support can answer “customer paid but order is pending” from one screen; Finance numbers identify their basis; every exception has an owner and resolution trail.

Product decisions

Decisions to make before implementation

These are product and operating decisions, not implementation trivia. Each answer affects schema, permissions, customer copy and support obligations.

  • Money flow: does every tenant use its own Razorpay account (recommended), or does ExSol collect and distribute funds? The latter creates marketplace, payout, tax and compliance work.
  • Currency: is INR-only acceptable for v1? If not, what determines foreign exchange, quote expiry and refund amount?
  • Methods: which methods are supported at launch: cash, Razorpay card/UPI, physical terminal, bank transfer? Which require provider verification versus staff evidence?
  • Booking deposits: are deposits credited to the balance, forfeited on no-show, refundable under policy, or convertible to account credit?
  • Expiry: how long does a payment-required appointment hold capacity? What happens if a verified payment arrives after the slot is released?
  • Refund authority: who can request, approve and execute a refund? Is a second approval required over a threshold?
  • Partial payment: may a customer pay deposit online and balance cash? Can a single payment cover more than one appointment/order in v1?
  • Tax and receipts: what invoice/receipt obligations apply to the target businesses, and does Finance need cash basis, accrual basis or both?
  • Data retention: how long are transaction references, webhook evidence and customer details retained? Which privacy/export/deletion requests must be supported?
  • Support: what is the script for provider outage, duplicate charge, “debited but pending,” late webhook and chargeback? Who owns the reconciliation queue?
Adopted security default: ExSol never receives or stores card number, CVV, UPI PIN or bank login. Hosted provider Checkout collects sensitive payment data. Provider secrets are per-tenant, encrypted at rest, write-only in settings and never placed in browser responses, audit detail or logs. A downloaded Test-mode CSV is a temporary secret-delivery mechanism, not application configuration: move its values into an ignored local/staging secret store, then remove the local download when it is no longer needed.
Adopted external-failure default: a signed provider event is stored and deduplicated before it changes business state. A duplicate becomes a no-op. A late event after a Booking hold expires never silently resurrects a potentially conflicting appointment; it opens a reconciliation case for staff. Provider failure leaves a visible pending request with a safe retry/expiry path.
Release bar

When can we call Payments production-ready?

Correctness

  • Every successful payment/refund has an immutable transaction and allocation.
  • Amount, currency, tenant and source ownership are server validated.
  • Duplicate/out-of-order webhook and browser retry behaviours are deterministic.
  • Partial payment and refund caps are correct.

Security

  • Raw webhook body and timing-safe per-tenant signature verification are tested.
  • Secrets are encrypted, write-only and rotation-ready.
  • Every handler follows enable gate then Owner bypass then permission matrix.
  • Public errors do not reveal another tenant/order.

Operations

  • Staff have receipt, timeline, pending/failed/late event and reconciliation views.
  • Expiry and reprocess jobs have clear idempotent behaviour.
  • Provider outage and manual recovery runbooks exist.
  • Refund result is observable, not assumed.

Platform quality

  • Module conforms to manifest/product/authz/mount/shared/CSS pattern.
  • Booking, POS and Orders use one payment core rather than copied logic.
  • Finance/Analytics use documented payment outcomes.
  • npm run typecheck, full npm test, docs generation and real-browser checks are green.

Do not start Razorpay integration until that review is agreed, the human coordinator assigns a migration number and the shared source-outcome contract is explicit.
