# POS v2.5 — Online Payment (Razorpay) Design Spec

**Date:** 2026-07-01
**Module:** POS (storefront)
**Status:** Drafted — awaiting user review. **Design-only**: no implementation plan or code until Razorpay API keys are available.
**Predecessor:** [`2026-06-29-pos-v2-storefront-design.md`](2026-06-29-pos-v2-storefront-design.md) — §11 sketched a "Razorpay seam"; this spec supersedes it (the seam's "zero schema change" assumption does not hold — see §3).
**Depends on:** nothing blocking. Coordinates migration numbering with the platform-branding chat (see §4).

---

## 1. Goal

Let a workspace owner accept **online payment (card / UPI) on the public storefront** via **Razorpay**, alongside the existing pay-on-pickup flow. Each tenant connects **their own Razorpay account** (per-tenant keys) so funds settle directly to the tenant — the platform never custodies money. The guest pays during checkout; **payment is confirmed server-side by a webhook**, never by the browser.

Success = a tenant pastes their Razorpay keys into settings, flips "online payment" on, and a guest can complete an order that lands in Sale History as `paid` without any staff action.

## 2. Scope

**In scope**

- Migration `051_pos_online_payment.sql` (see §4): per-method payment toggles + per-tenant Razorpay credentials on `public.clients`; `razorpay_order_id` + `razorpay_payment_id` on `public.sales`.
- New shared helper `netlify/functions/_shared/crypto.ts`: AES-256-GCM encrypt/decrypt of tenant secrets with a platform key-encryption-key (KEK) from env.
- New shared helper `netlify/functions/_shared/razorpay.ts`: thin Razorpay REST client (create order) + webhook HMAC verify, using a tenant's decrypted keys.
- New authed endpoint `PATCH /api/client-settings/payments`: set `pickupEnabled`, `onlineEnabled`, and Razorpay credentials (`keyId`, `keySecret`, `webhookSecret`). Secrets are **write-only** (never returned).
- New authed endpoint `GET /api/client-settings/payments`: returns toggles + a `razorpayConfigured: boolean` (never the secrets).
- Extend `pub-menu.ts` response with the tenant's **enabled payment methods** so the storefront knows what to offer.
- Extend `pub-sale-create.ts`: for online orders, create the sale `pending_payment`, create a Razorpay order server-side with the tenant's keys, persist `razorpay_order_id`, and return `{ sale, razorpay: { orderId, keyId, amountPaise, currency } }`.
- New public endpoint `pos-razorpay-webhook.ts`: HMAC-verified, tenant-resolved, advances the sale to `paid`.
- FE: storefront details page gains a payment-method choice (when both methods are enabled) and the Razorpay Checkout integration; a settings **Payments card** for the owner.

**Out of scope** (explicit)

- **Turnstile bot defense → v2.6** (honeypot + IP rate-limit from v2 still guard `pub-sale-create`).
- **Refunds via Razorpay.** Staff refund/void stays the existing manual FSM transition; no money is moved through Razorpay in v2.5.
- Razorpay **Route / connected accounts / platform commission** (per the money-flow decision — per-tenant keys, no marketplace).
- Saved cards, partial payments, multi-currency (INR only in v1), tips.
- **Auto-expiry of abandoned online orders** (they linger as `pending_payment`; staff can cancel — see §7.4). A cron sweep is a follow-up.

## 3. Decisions

| Decision | Choice | Reasoning |
|---|---|---|
| Money flow | **Per-tenant Razorpay keys** | Funds settle directly to the tenant; platform never holds money or owns reconciliation. Each tenant connects their own Razorpay account. |
| Payment-method config | **Two independent booleans** `storefront_pickup_enabled` + `storefront_online_enabled` | Yields Both (guest chooses) / One (either) / None (browse-only menu). Cleaner than a single enum and directly models "offer both, one, or none". |
| Master switch | `storefront_enabled` unchanged | Still the top-level on/off. If it's off, nothing below matters. If on but both payment toggles off → menu renders read-only (no cart/checkout). |
| Secret storage | **AES-256-GCM**, KEK from env `RAZORPAY_ENC_KEY` (32 raw bytes, base64 in env) | `key_secret` + `webhook_secret` are ciphertext at rest. `key_id` is public-ish (it ships to the browser for Checkout) → stored plaintext. Secrets are never returned by any endpoint. |
| Order timing | **Create sale `pending_payment` first, then the Razorpay order; webhook confirms** | The receipt page needs a sale to poll immediately. Matches the existing `pending_payment` semantics and reuses `applyTransition`. Abandonment leaves a harmless pending sale (§7.4). |
| Webhook lookup key | New `razorpay_order_id` column (indexed) | `payment_ref` is **already used** for idempotency (`idem:<key>`), so it can't double as the Razorpay order id. This is why v2.5 needs a migration (the v2 "zero-schema seam" was wrong on this point). |
| Amount integrity | Server-computed total in **paise**; webhook re-checks captured amount == total | The browser only ever carries an opaque `orderId`. Price cannot be moved client-side. |
| Idempotency (create) | Keep v2's `payment_ref = 'idem:'+key` dedup | A double-submit reuses the same sale instead of creating a second Razorpay order. |
| Idempotency (webhook) | No-op if the sale is already `paid` | Razorpay may deliver a webhook more than once; the transition must be safe to replay. |
| Migration number | **`051`** (coordinate) | Prod is at `049`; the platform-branding chat plans `050`. Confirm `050` is taken before locking `051` (see §4). |
| Checkout mechanism | Razorpay **Checkout.js** (`checkout.razorpay.com/v1/checkout.js`), loaded lazily on the details page | Standard hosted checkout; PCI scope stays with Razorpay. External script is acceptable for a payment flow. |
| Currency | INR only | Matches the rupee-denominated `*_cents` columns (stored as paise). |

## 4. Data model — migration 051

> **Coordination:** verify `050` is claimed by the platform-branding chat before locking this as `051`. Migrations key on the full filename, so a same-number collision is survivable but confusing — pick the next genuinely free integer at implementation time.

```sql
-- migration: 051_pos_online_payment.sql
--
-- Adds per-method storefront payment toggles + per-tenant Razorpay credentials
-- (secrets encrypted at rest by the app; see _shared/crypto.ts), and the
-- Razorpay order/payment identifiers needed to reconcile a webhook to a sale.

ALTER TABLE public.clients
  ADD COLUMN IF NOT EXISTS storefront_pickup_enabled boolean NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS storefront_online_enabled boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS razorpay_key_id           text,
  ADD COLUMN IF NOT EXISTS razorpay_key_secret_enc   text,   -- AES-256-GCM ciphertext (iv:tag:ct, base64)
  ADD COLUMN IF NOT EXISTS razorpay_webhook_secret_enc text; -- AES-256-GCM ciphertext (iv:tag:ct, base64)

ALTER TABLE public.sales
  ADD COLUMN IF NOT EXISTS razorpay_order_id   text,
  ADD COLUMN IF NOT EXISTS razorpay_payment_id text;

-- Webhook resolves a sale by the Razorpay order id; make that lookup fast + unique.
CREATE UNIQUE INDEX IF NOT EXISTS sales_razorpay_order_id_key
  ON public.sales (razorpay_order_id)
  WHERE razorpay_order_id IS NOT NULL;
```

Column semantics:

- `storefront_pickup_enabled` defaults **true** — preserves current behavior (existing storefronts already take pay-on-pickup orders). `storefront_online_enabled` defaults **false** — online is opt-in and requires credentials.
- The two `*_enc` columns store a self-describing ciphertext string `iv:tag:ciphertext` (all base64). Null = not configured.
- `razorpay_key_id` is plaintext (it is sent to the browser for Checkout). Null = not configured.
- A tenant is "online-ready" iff `storefront_online_enabled AND razorpay_key_id IS NOT NULL AND razorpay_key_secret_enc IS NOT NULL AND razorpay_webhook_secret_enc IS NOT NULL`. The settings PATCH enforces this (§5.2) so `onlineEnabled=true` can't be saved without complete credentials.
- `sales.razorpay_order_id` unique-partial index doubles as a guard against two sales sharing an order id.
- `payment_method` (`'cash' | 'razorpay'`) and `channel` already exist from v1/v2; online orders use `payment_method='razorpay'`, `channel='online'`.

## 5. Backend

### 5.1 `netlify/functions/_shared/crypto.ts`

AES-256-GCM using Node's `crypto` (available in Netlify Functions). The KEK is read once from `process.env.RAZORPAY_ENC_KEY` (base64 of 32 random bytes).

```ts
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';

function kek(): Buffer {
  const raw = process.env.RAZORPAY_ENC_KEY;
  if (!raw) throw new Error('RAZORPAY_ENC_KEY not set');
  const key = Buffer.from(raw, 'base64');
  if (key.length !== 32) throw new Error('RAZORPAY_ENC_KEY must decode to 32 bytes');
  return key;
}

/** Returns "iv:tag:ciphertext", all base64. */
export function encryptSecret(plaintext: string): string {
  const iv = randomBytes(12);
  const c = createCipheriv('aes-256-gcm', kek(), iv);
  const ct = Buffer.concat([c.update(plaintext, 'utf8'), c.final()]);
  const tag = c.getAuthTag();
  return [iv.toString('base64'), tag.toString('base64'), ct.toString('base64')].join(':');
}

export function decryptSecret(blob: string): string {
  const [ivB64, tagB64, ctB64] = blob.split(':');
  if (!ivB64 || !tagB64 || !ctB64) throw new Error('malformed ciphertext');
  const d = createDecipheriv('aes-256-gcm', kek(), Buffer.from(ivB64, 'base64'));
  d.setAuthTag(Buffer.from(tagB64, 'base64'));
  return Buffer.concat([d.update(Buffer.from(ctB64, 'base64')), d.final()]).toString('utf8');
}
```

Notes: GCM's auth tag makes tampering detectable (decrypt throws). Rotating `RAZORPAY_ENC_KEY` invalidates all stored secrets — document as ops knowledge; re-entry by tenants is the recovery path (v1 has no key-rotation re-encrypt job).

### 5.2 `PATCH /api/client-settings/payments` (authed) + `GET`

Auth: `authenticateForPermission(req, '_platform.settings.edit')` with L1 bypass via `requirePermission`; `resolveClientIdOrRespond` → `clientId`.

**GET** returns:
```ts
{ pickupEnabled: boolean, onlineEnabled: boolean, razorpayConfigured: boolean, razorpayKeyId: string | null }
```
`razorpayKeyId` is safe to return (public). Secrets are never returned; `razorpayConfigured` reflects presence of all three credential columns.

**PATCH** body (all optional; only supplied fields change):
```ts
{
  pickupEnabled?: boolean;
  onlineEnabled?: boolean;
  razorpayKeyId?: string | null;       // plaintext
  razorpayKeySecret?: string | null;   // encrypted before write; null clears
  razorpayWebhookSecret?: string | null;
}
```
Validation (Zod strict):
1. If the resulting state would have `onlineEnabled=true` but the credential set is incomplete (after applying this patch on top of current DB values) → `400 online_requires_credentials`. This prevents enabling online without working keys.
2. Secrets are `encryptSecret(...)`-ed before the UPDATE; `null` writes SQL `NULL` (clears). Non-secret fields set directly.
3. `logAudit({ op: 'client.payments_updated', clientId, detail: { fields_changed, razorpay_reconfigured } })` — **never log secret values**.

Errors: `400 validation_failed | online_requires_credentials`, `401`, `403`, `405`.

### 5.3 `pub-menu.ts` extension

Add a `payments` object to the existing public menu response so the storefront FE knows what to offer without a second call:
```ts
payments: { pickup: boolean, online: boolean }   // online == tenant is "online-ready" (§4)
```
No secrets, no key id here (the key id is returned only from `pub-sale-create` when an online order is actually placed — minimizes exposure surface).

### 5.4 `pub-sale-create.ts` extension (online path)

Request gains an optional `paymentMethod: 'pickup' | 'online'` (default `'pickup'`). Existing honeypot, IP rate-limit, idempotency, and **server-side price snapshot** all stay.

Flow when `paymentMethod === 'online'`:
1. Reject with `400 online_not_available` if the tenant is not online-ready (§4).
2. Compute the authoritative total from the server price snapshot (existing logic), in paise.
3. Insert the sale `pending_payment`, `channel='online'`, `payment_method='razorpay'`, creator NULL (guest), `source='storefront'` — same as the pickup path but with the online channel/method.
4. Create a Razorpay **order** via `_shared/razorpay.ts` using the tenant's decrypted `key_secret`: `amount = totalPaise`, `currency = 'INR'`, `receipt = sale.id`, `notes = { saleId, clientId }`.
5. `UPDATE sales SET razorpay_order_id = <order.id> WHERE id = <sale.id>`.
6. Respond `201 { sale, razorpay: { orderId, keyId, amountPaise, currency: 'INR' } }` (`keyId` = plaintext `razorpay_key_id`).

Pickup path is unchanged (`201 { sale }`, no `razorpay`).

Failure handling: if the Razorpay order call fails (step 4), the sale row already exists as `pending_payment` with no `razorpay_order_id`. Return `502 payment_provider_error`; the FE shows a retryable error. The dangling pending sale is harmless (§7.4). (We intentionally don't roll back the sale insert — keeping it simple and idempotent via the existing `payment_ref` dedup; a retry with the same idempotency key reuses the sale.)

### 5.5 `pos-razorpay-webhook.ts` (public, HMAC-verified)

`export const config = { path: '/api/pos/razorpay-webhook', method: 'POST' }`.

1. Read the **raw body** (needed for HMAC — do not parse first).
2. Parse just enough to get the Razorpay order id (`payload.payment.entity.order_id` for `payment.captured`, or `payload.order.entity.id` for `order.paid`).
3. Resolve the sale by `razorpay_order_id` → its `bucket_id` → the tenant's `razorpay_webhook_secret_enc` → `decryptSecret`. If no sale / not online-ready → `404` (no information leak).
4. Verify `X-Razorpay-Signature` == `HMAC-SHA256(rawBody, webhookSecret)` using a **timing-safe compare**. Mismatch → `401 invalid_signature`.
5. Idempotency: if the sale is already `paid` (or beyond) → `200 { ok: true }` no-op.
6. On `payment.captured` / `order.paid`: verify `payload…amount === sale.total_cents` (paise). Mismatch → `400 amount_mismatch` (and audit-log it — a red flag).
7. Persist `razorpay_payment_id`, then `applyTransition(sale, 'markPaid', { paymentMethod: 'razorpay' })` → `paid`. Channel `online` means **no auto-fulfill** (pickup/fulfillment stays a staff step, consistent with v2).
8. `200 { ok: true }`.

Only `payment.captured` and `order.paid` events act; everything else → `200` ignore (so Razorpay doesn't retry). Never trust the event's client-controllable fields beyond the signed payload.

### 5.6 `netlify/functions/_shared/razorpay.ts`

```ts
// createOrder(keyId, keySecret, { amountPaise, receipt, notes }) -> { id, amount, currency }
//   POST https://api.razorpay.com/v1/orders  (HTTP Basic: keyId:keySecret)
// verifyWebhookSignature(rawBody, signatureHeader, webhookSecret) -> boolean  (HMAC-SHA256, timingSafeEqual)
```
Thin `fetch` wrapper; no SDK dependency (keeps `external_node_modules` untouched). Basic-auth header built from the tenant's decrypted keys per request (functions don't share memory — never cache a tenant's secret across invocations).

### 5.7 Function config / routing

All new functions are **flat files** under `netlify/functions/` with explicit `config.path` + `config.method` (per the routing-collision + subdir-discovery memory notes). Post-deploy, probe each new endpoint for the new-function-404 trap → `netlify api restoreSiteDeploy` if needed.

## 6. Frontend

### 6.1 Details page — payment choice (`StorefrontDetailsPage.tsx`)

- Read `payments` from the loaded menu (§5.3).
- If `pickup && online` → a two-option selector ("Pay now" / "Pay at pickup"), default "Pay now".
- If only one enabled → no selector; that method is implicit.
- If neither → the storefront is browse-only; the checkout route is not reachable (guarded at the menu/cart level too).
- On submit, `pub-sale-create` is called with the chosen `paymentMethod`.

### 6.2 Razorpay Checkout integration

- Lazy-load `checkout.razorpay.com/v1/checkout.js` on first online checkout (a small `loadRazorpay()` promise-caching helper).
- On a `201` with `razorpay`, open `new window.Razorpay({ key: keyId, order_id: orderId, amount: amountPaise, currency, name: tenantName, prefill: { name, email, contact }, handler, modal.ondismiss })`.
- `handler` (client-side success callback) is **advisory only** — it just navigates to the receipt route (`/menu/:slug/order/:saleId`). The receipt page polls (existing 20s poll) and reflects `paid` once the **webhook** processes. We never mark paid from the browser.
- `modal.ondismiss` → stay on the details page with a "Payment cancelled — your order is saved as pending" note (the sale exists as pending; §7.4).
- Errors from `pub-sale-create` (`online_not_available`, `payment_provider_error`) → inline retryable message.

### 6.3 Settings — Payments card

- Mounts alongside the existing storefront settings (owner / `_platform.settings.edit` gated, with L1 bypass).
- Controls: pickup toggle; online toggle; Razorpay credentials form (`keyId`, `keySecret`, `webhookSecret`) with a "configured ✓" state (never shows stored secrets — inputs are blank; submitting blank leaves them unchanged, an explicit "clear" clears).
- Disables the online toggle until credentials are saved; surfaces the server's `online_requires_credentials` error inline.
- Shows the webhook URL to paste into the Razorpay dashboard: `https://<site>/api/pos/razorpay-webhook`.

### 6.4 Receipt page

Unchanged — it already polls and renders the current status. Online orders simply transition `pending_payment → paid` when the webhook lands, which the poll picks up.

## 7. Security & correctness

1. **Secrets at rest:** AES-256-GCM; KEK in env only; never returned by any endpoint; never logged. GCM tag detects tampering.
2. **Webhook authenticity:** per-tenant HMAC-SHA256 over the raw body, timing-safe compare; tenant resolved from the signed order id → its own webhook secret.
3. **Amount integrity:** total is server-computed from the price snapshot; the webhook re-verifies the captured amount equals the stored total; the browser carries only an opaque order id.
4. **Abandonment:** an unpaid online sale stays `pending_payment` (no `razorpay_payment_id`), visible in Sale History with the Storefront badge; staff can cancel via the existing FSM. Auto-expiry is a follow-up (§10).
5. **Idempotency:** create-path dedup via `payment_ref='idem:'+key`; webhook is a no-op once `paid` (Razorpay redelivers).
6. **Rate-limit / bots:** existing honeypot + IP rate-limit on `pub-sale-create` remain the guard until Turnstile (v2.6).
7. **Leak guards:** webhook and public endpoints return `404`/generic errors rather than confirming existence of a tenant/sale; receipt bearer-token guard from v2 unchanged.
8. **PCI:** card data never touches our servers — Razorpay Checkout collects it. Our scope is SAQ-A-ish.

## 8. Testing

### 8.1 Unit (~10)
- `crypto`: round-trip encrypt→decrypt; wrong-key decrypt throws; tampered ciphertext (flip a tag byte) throws; missing/short KEK throws.
- `razorpay.verifyWebhookSignature`: correct sig → true; wrong sig → false; timing-safe path exercised.
- Details-page payment selector: both → selector shown, one → implicit, none → no checkout.
- `loadRazorpay()` caches the script promise (one injection under concurrent calls).

### 8.2 Integration (~14)
- Migration `051` adds all columns + the partial unique index.
- `PATCH /payments`: sets toggles; `onlineEnabled=true` without full creds → `400 online_requires_credentials`; secrets are stored encrypted (assert DB value != plaintext and decrypts back); `GET` never returns secrets and reports `razorpayConfigured` correctly; no-auth → 401; wrong-perm → 403.
- `pub-menu`: `payments` reflects the toggles; online false when creds missing even if toggle true.
- `pub-sale-create` online: online-ready tenant → `201` with `razorpay.{orderId,keyId,amountPaise}` and a `pending_payment` sale carrying `razorpay_order_id`; not-online-ready → `400 online_not_available`; provider failure (mock 500) → `502` with the sale still pending; idempotent re-submit reuses the sale (one Razorpay order).
- `pos-razorpay-webhook`: valid sig + `payment.captured` matching amount → sale `paid`, `razorpay_payment_id` set; bad sig → `401`; unknown order id → `404`; amount mismatch → `400 amount_mismatch`; **redelivery** (same event twice) → second is a `200` no-op, still single `paid`; non-actionable event → `200` ignore.

> Test-DB note (per memory): tests share a persistent dev DB with no teardown. Randomize unique-constrained literals (slugs, order ids) and keep seeded rows on distinct ranges so re-runs don't collide. Run the FULL suite before declaring green.

### 8.3 Not tested in v1
- Live Razorpay Checkout UI (browser-smoke only, needs real keys).
- Turnstile (v2.6). Refunds. Auto-expiry cron.

## 9. Deployment

- **New env var `RAZORPAY_ENC_KEY`** (base64 of 32 random bytes) must be set in **every** Netlify context that runs functions (per the deploy-checklist memory: per-context env coverage). Absent → the payments endpoints throw on encrypt/decrypt (fail-closed).
- Migration `051` is additive → code-first or migration-first both safe. Run against the **prod** Neon branch before promoting code that reads the new columns (per the migrate-before-promote memory).
- No new npm deps (Razorpay via `fetch`; Checkout.js is a browser script tag).
- Post-deploy: probe the 3 new endpoints for the new-function-404 trap; `restoreSiteDeploy` if any 404.
- Per-tenant Razorpay dashboard setup (tenant-side, documented in the settings card): set the webhook URL to `/api/pos/razorpay-webhook`, subscribe to `payment.captured` (+ `order.paid`), and paste the webhook secret into settings.
