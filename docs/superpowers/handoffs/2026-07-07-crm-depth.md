## Gotchas / decisions

- **crm_customers is a derived read-model** (materialized via `refreshCustomers`). It has **no FK** to
  sales/bookings — dashboard/timeline/repeat-cart bridge by **normalized-phone last-10 digits** (crm phone
  is canonical `+91…`, sales/bookings store raw input) or **lowercased email**. Same bridge v1's
  customer-detail already used. Money is BIGINT (string from Neon) → `Number()`.
- **Timeline** merges 5 sources: crm_notes + campaign_sends (direct `customer_id` FK), email_outbox
  (matched by `to_email` — no FK, lossy if the customer changed email), sales + bookings (phone/email).
- **Lead capture is PUBLIC.** Guarded exactly like the storefront: honeypot (filled ⇒ silent 200),
  per-IP + per-slug Blobs rate limit, and a tenant resolver that 404s unless CRM is enabled — never
  leaking slug existence. **Rate-limit uses Netlify Blobs → tests mock `@netlify/blobs`.**
- **Convert** materializes a crm_customer via the read-model dedupe key (`phone:<normalized>` |
  `email:<lower>`) with **source='storefront'** (the source CHECK is pos/storefront/booking — a web lead
  is closest to storefront; no CHECK change needed). Only a `new` lead can be acted on (409 otherwise).
- **Social sync is a MOCK seam** (`lib/social-import.ts`): connect sets a synthetic label; import
  synthesizes a deterministic batch (offset by `imported_total`, so re-imports don't duplicate) into
  `crm_leads` (source='social') → they appear in the Leads inbox. Real OAuth/keys are the only follow-up;
  the endpoint/UI stay identical. Mirrors marketing's social pattern.
