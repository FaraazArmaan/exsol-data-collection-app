ExSol Suite

A unified multi-tenant SaaS platform that produces seven distinct customer-facing products from a single codebase and shared data model: an internal ERP, a data-collection onboarding app, a booking system, an ecommerce storefront, an in-store POS, a catalog showcase site, and per-brand portfolio sites. This document covers the High-Level and Low-Level System Design as they exist on the deployed codebase.

Stack: React 18 · Vite · TypeScript · Netlify Functions v2 · Neon Postgres · One repo, seven outputs

For a more accessible, in-depth walkthrough (suitable for non-developer readers and longer onboarding), see architecture (expansive).

The Suite

Seven products, one codebase

The whole platform is a single Git repo. From that one tree, the build produces multiple shippable products that share authentication, data, code, and operational tooling. The architectural payoff is that a fix to (say) the product catalog improves the ecommerce storefront, the POS, the catalog site, AND the data-collection onboarding flow simultaneously — because they all read from the same module.

The Internal App

Overview

What we are building, restated

ExSol is a per-workspace operational platform. Each Client (e.g. papa-s-saloon) gets an isolated workspace inside the Internal App with team management, file storage, product catalog, point-of-sale, and booking. The Internal App is the operator surface; the six customer-facing products are configurable views into the same data. Two main authentication audiences inside the Internal App: a cross-workspace Admin console for operational oversight, and a per-workspace Owner / Staff portal for day-to-day work. Customer-facing products bypass auth entirely via opt-in tenant flags.

High-Level System Design

The macro picture: layers of the system, what they own, and how requests flow through them.

1.1 System Context §context

ExSol sits between end-users on the open web and a managed Postgres backend, with Netlify acting as both CDN and serverless function host. There is no separate API server, no Kubernetes cluster, no message bus — the entire backend is a flat set of HTTP-triggered functions.

Actors

  • End-users (Owners, Staff) — log into a workspace at /c/<slug>
  • Admins — operational console at / (root)
  • Public guests (POS v2) — checkout at /c/<slug>/menu/:slug

External services

  • Google OAuth — sign-in + workspace invite
  • Netlify Blobs — image storage with thumbnail pipeline
  • Razorpay (planned) — POS payment HMAC webhook

1.4 Multi-Tenancy §multitenancy

Every meaningful table carries client_id and is scoped at query time. There is no Postgres Row-Level Security — tenancy is enforced in application code via the auth helper layer.

Two session kinds

  • admin — global across all clients; identified by admins.id; cross-client by definition.
  • bucket_user — pinned to a single client_id in the JWT; cannot reach other clients even with a forged URL.

Cross-client guard

resolveClientId(session, req) checks the URL's ?client=<uuid> param against the session's pinned client. Bucket-user mismatch → 403 forbidden_cross_client. Admins must explicitly pass a ?client=; bucket-users may not.

1.5 Module Registry §modules

Modules and Products are declared statically in src/modules/registry/. Each Client enables individual Products via the client_enabled_products table, which transitively unlocks Modules and their permission keys. 

export const productRegistry = {
  'saloon-booking': saloonBookingProduct,
  'products':       productsProduct,
  'pos':            posProduct,    // requires: ['products']
} as const;

Two permission model classes coexist (see §2.2 below). Most Modules expose data_buckets + CRUD verbs (the matrix model). POS introduces action-namespace permissions (pos.menu.view, pos.sale.markPaid, …) where the matrix doesn't fit.

1.6 Deployment Topology §deployment

Git-driven, no manual deploys, no CI-runner-managed credentials beyond Netlify env vars.

Migrations are out-of-band from code deploys (manual via npm run migrate); design assumes additive-first migrations safe to run before code.

1.7 Customer-Facing Surfaces & URL Routing §cust-surfaces

Workspaces serve two distinct kinds of pages: staff/owner pages (require login, gated by RBAC) and customer-facing pages (open to anonymous users — POS storefront ordering, booking widget). The URL structure differs between these surfaces, and clients have three options for how customers reach them: on the Exsol shared domain, on a branded subdomain, or on the client's own domain.

URL hosting patterns

Customer-facing surfaces can be reached three ways. Each has a different operator setup cost vs branding outcome.

All three patterns terminate at the same workspace-resolver helper. Host header beats path; path is the fallback.

Request resolution rules

// Pseudo — netlify/functions/_shared/resolve-workspace.ts (planned for v2)
export async function resolveWorkspace(req: Request): Promise<{ slug: string, clientId: string }> {
  const host = req.headers.get('host') ?? '';
  const url  = new URL(req.url);

  // 1. Custom domain? Look up alias list
  if (!host.endsWith('exsoldatacollectionapp.netlify.app') && !host.endsWith('.exsol.app')) {
    const [c] = await sql`SELECT id, slug FROM clients WHERE ${host} = ANY(domain_aliases) LIMIT 1`;
    if (c) return { slug: c.slug, clientId: c.id };
  }

  // 2. Wildcard subdomain? Leftmost label = slug
  const subMatch = host.match(/^([a-z0-9-]+)\.exsol\.app$/);
  if (subMatch) return resolveBySlug(subMatch[1]);

  // 3. Fallback: path-based — /menu/:slug, /book/:slug/:svc, /c/:slug/...
  const seg = url.pathname.split('/').filter(Boolean);
  const pathSlug = seg[0] === 'c' || seg[0] === 'menu' || seg[0] === 'book' ? seg[1] : null;
  if (pathSlug) return resolveBySlug(pathSlug);

  throw new NotFound();
}

Trade-offs

PatternSetup costBrandingSEOWhen to recommend
A · Path None Low (Exsol brand visible) Low (shared domain authority) Trial, signup, first-touch
B · Subdomain One-time platform: wildcard cert + DNS Mid (workspace name visible) Mid (subdomain inherits some authority) Default for paid plans
C · Custom domain Per-client: CNAME + Netlify alias entry High (no Exsol footprint) High (client owns domain authority) Higher tier; existing brand

Operator setup — Pattern C (custom domain)

The end-to-end flow for a client (e.g. papa's saloon) to point papas-saloon.com at their Exsol workspace:

  1. Client side: add a CNAME record at their DNS provider: papas-saloon.com → exsoldatacollectionapp.netlify.app
  2. Exsol side: add the domain to the Netlify site's domain aliases (one-line API call or UI click)
  3. Netlify: auto-issues a Let's Encrypt cert (~minutes)
  4. Workspace: add papas-saloon.com to clients.domain_aliases TEXT[] on the matching client row (planned schema change for v2 storefront)
  5. Verify: hit https://papas-saloon.com/menu — resolver sees the host, looks up the alias, returns workspace identity, normal request flow proceeds
Why this works without code surgery
The whole codebase already scopes every query by client_id. Adding host-based resolution is purely additive: a new helper that picks the same client_id we'd have got from the path, then passes it to the existing perms/data layer unchanged. No refactor of any handler is needed.
Edge case — cross-workspace assets
Product images stored in Netlify Blobs are addressed by product-images/<client_id>/<product_id>/<blob_id>. If a workspace embeds a customer URL that points to another workspace's product (e.g. via a misconfigured affiliate link), the resolver picks the host workspace but the blob path still encodes the original workspace's client_id — so the image fetch will 404 unless explicitly cross-shared. This is the right default.
Part 2

Low-Level System Design

How things actually work — the auth chain, table relationships, state machine, and operational patterns that hold the system together.

2.1 Auth Flow §auth-flow

Every protected endpoint goes through the same chain. The chain has two entry points (admin or bucket-user) and a clean L1-Owner fast-path that's identical across modules. 

// netlify/functions/_shared/permissions.ts (simplified)
export async function requirePermission(req: Request, key: string): Promise<Session> {
  // 1. Try admin session first
  try { return { kind: 'admin', ...await requireAdmin(req) }; }
  catch (e) { if (!(e instanceof UnauthorizedError)) throw e; }

  // 2. Fall through to bucket-user via JWT cookie
  const claims = await verifyBucketUserSession(readBuCookieToken(req));
  const { level_number, client_id } = await sql`SELECT ... FROM user_nodes WHERE id = ${claims.sub}`;

  // 3. L1 (Primary Owner) bypasses the matrix check
  if (level_number === 1) return { kind: 'bucket_user', level_number: 1, client_id, ... };

  // 4. Otherwise, consult the per-level permission matrix
  const matrix = await getLevelMatrix(client_id, level_number);
  if (!matrix[key]) throw new ForbiddenError(key);
  return { kind: 'bucket_user', level_number, client_id, ... };
}
POS-specific authz
POS uses a parallel helper requirePos(req, requiredKeys[]) that adds two extras: a 412 products_module_required / pos_module_not_enabled dependency gate BEFORE the perm check, and a full perms set returned in the context (so the FSM action buttons can render per-action). The L1 fast-path is identical.

2.2 Permission Model §perms

Two distinct permission models coexist in one matrix table (client_levels.permissions JSONB). Choosing the right model for a feature depends on whether the operations are CRUD on a resource (bucket/verb) or state-machine transitions / API actions (action-namespace).

Bucket / Verb model

For typical CRUD modules. Permissions are <module>.<bucket>.<verb>.

"users.users.create": true,
"users.users.edit":   true,
"users.users.delete": false,
"products.products.create": true,
Used by Files Products Team

Action-namespace model

For state-machine ops where "verbs on a resource" doesn't carve cleanly. Permissions are <module>.<action-path>.

"pos.menu.view":     true,
"pos.sale.create":   true,
"pos.sale.markPaid": true,
"pos.sale.refund":   false,
"pos.history.viewAll": false,
Used by POS
L1 Owner = all-on by default
Both models honor level_number === 1 as an implicit grant of every key, enforced at the helper layer (requirePermission, requirePos) and mirrored in the FE gates (Sidebar, PosRouteMounts). L2+ levels require explicit grants via the Access Level admin dashboard, which now renders both models (matrix grid for bucket/verb, per-action toggles for action-namespace).

Key design decisions
  • Snapshot prices in sale_lines — later product edits don't rewrite history.
  • Per-bucket order_no — sequential within a workspace, not global; allocated via MAX+1 with 5-retry on 23505.

2.4 POS Sale State Machine §fsm

Sale states are enforced server-side. Each transition checks both the FSM legality (current → target valid?) AND the caller's permission for the target action. The instore channel collapses paid→fulfilled in one call (auto-fulfill, recorded as detail.auto = true in audit).

Each transition requires the caller to hold the corresponding pos.sale.<action> permission AND the FSM source state to be valid.

2.5 Booking Module Flow §booking

Booking lets clients sell time on named resources (a stylist, a chair, a meeting room) with per-service rules — duration, per-weekday hours, optional payment up front. Two surfaces meet at the same backend: a public customer widget that auto-registers users by phone+email, and a staff calendar that manages the catalog and reads the booking grid. The core invariant the design protects is no double-booking: a single resource cannot be reserved for overlapping time ranges, enforced atomically by the database.

Customer journey (public widget)

The customer never directly contacts the database — every step is mediated by a public API endpoint with rate limiting + honeypot.

No-overbooking invariant

The hard constraint is enforced at the schema level using a PostgreSQL GIST exclusion constraint rather than application-side checks. This makes it atomic — concurrent requests for the same slot can't both succeed: 

-- bookings (planned schema for v1)
CREATE TABLE bookings (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  client_id    uuid NOT NULL REFERENCES clients(id),
  resource_id  uuid NOT NULL REFERENCES resources(id),
  service_id   uuid NOT NULL REFERENCES services(id),
  customer_user_node_id uuid REFERENCES user_nodes(id),  -- nullable for guest holds
  time_range   tstzrange NOT NULL,
  status       booking_status NOT NULL DEFAULT 'pending_payment',
  ...,
  -- atomicity: two bookings on the same resource cannot overlap
  EXCLUDE USING GIST (
    resource_id WITH =,
    time_range  WITH &&
  ) WHERE (status IN ('pending_payment', 'confirmed'))
);
Why GIST + WHERE clause
Cancelled or refunded bookings shouldn't block the slot — the partial index condition lets only active bookings hold the slot. A retry after a webhook timeout (still pending_payment) is still safe: the conflicting row IS the original, and the idempotency key on (client, phone, time_range) catches the re-submission.

Per-service payment policy

Each service declares a payment mode that drives the FSM at booking time. Resource-time holds are kept short for free services and revert to free-for-grabs if payment doesn't complete in the configured window.

Service modeInitial booking statusSlot held untilBecomes
payment: 'none' confirmed start time completed at end_time or staff marks no-show
payment: 'optional' confirmed start time same as 'none'; payment can be taken at the counter
payment: 'required' pending_payment +10 min OR webhook confirmed on Razorpay webhook · or cancelled on timeout

Staff side (calendar surface)

The owner / staff calendar is a workspace-portal page (under /c/<slug>/booking) gated on the standard bucket/verb permission model. Three primary views:

  • Day grid — vertical time axis × horizontal resources. Color-coded by status. Click a cell to view/edit a booking.
  • Service catalog — CRUD for services, duration, payment policy, per-weekday hours.
  • Resource manager — CRUD for named resources, optionally tied to a staff user_node for "the stylist who's logged in only sees their column".
Timezone handling
All time_range values are stored as tstzrange (timestamp WITH time zone). Conversion to the workspace's display timezone happens at the FE / formatter layer only — storage and the GIST overlap check are always in UTC, so a daylight-saving boundary can't accidentally create or hide an overlap.

2.6 API Patterns §api-patterns

Every function file follows the same skeleton. Consistency at this layer makes the ~65 endpoints feel like one codebase.

// netlify/functions/<name>.ts
import type { Context } from '@netlify/functions';
import { jsonError, jsonOk } from './_shared/http';
import { authenticateForPermission } from './_shared/permissions';
import { logAudit } from './_shared/audit';
import { db } from './_shared/db';

export const config = { path: '/api/foo', method: 'POST' };

export default async (req: Request, _ctx: Context) => {
  if (req.method !== 'POST') return jsonError(405, 'method_not_allowed');

  // 1. Auth + perm gate (returns Response on failure)
  const auth = await authenticateForPermission(req, 'foo.bar.create');
  if (auth instanceof Response) return auth;

  // 2. Resolve tenant scope
  const scope = resolveClientIdOrRespond(auth, req);
  if (scope instanceof Response) return scope;

  // 3. Validate body with zod
  const body = FooCreateBody.safeParse(await req.json());
  if (!body.success) return jsonError(400, 'invalid_body');

  // 4. Do the work
  const sql = db();
  const [row] = await sql`INSERT INTO foos (...) VALUES (...) RETURNING *`;

  // 5. Audit
  await logAudit(auth, 'foo.created', { target_id: row.id, detail: { ... } });

  return jsonOk(row);
};

2.7 Audit & Migrations §audit

Audit trail

Every mutation writes a row to audit_log via logAudit(session, op, { target, detail }). The detail JSONB is opt-in: include enough to reconstruct intent (e.g., { from, to, auto, reason } on FSM transitions; { byte_count, table_counts } on exports).

Indexed by (client_id, occurred_at) for cheap timeline queries per workspace.

Migration system

Sequential numbered files in db/migrations/<NNN>_<name>.sql. Tracked in schema_migrations(version, applied_at). Apply with npm run migrate (dev) or DATABASE_URL=$PROD_DB_URL npm run migrate (prod).

Dry-run: npm run migrate -- --status. Idempotent on re-run via the version table — applied migrations are skipped.

Order of operations
Additive migrations: apply to prod BEFORE pushing code that depends on the new column/table.
Destructive migrations: push code FIRST (removing references), THEN apply the destructive migration. Order is inverted.
