Why does Workforce not own login identity?

user_nodes and the platform auth system own identity, session, and permission truth. Workforce owns employment-specific facts and operational records. This avoids two employee accounts drifting apart and keeps one login/security model for the whole platform.

Why do Project Service and Workforce share auth?

The product unlocks both systems and they use the same tenant/user hierarchy. Sharing _workforce-authz.ts means a staff member cannot pass one set of scope rules in Projects and a different set in Workforce.

One real journey: an employee clocks out

Screen: the self-service workforce page shows only temporary browser state and calls a typed shared API function.
HTTP boundary: workforce-me-clock-out.ts resolves the signed-in worker, checks the Workforce product and permission/scope rules, and validates the request.
Domain rule: the self-time helper confirms there is an open time record, applies break/time policy, and rejects an invalid transition rather than guessing.
Durable truth: the time ledger/punch record is written tenant-scoped, with audit-ready timestamps. Later timesheets, payroll export, reporting, and manager review read that same truth.

Good, Bad, Missing

Good

  • Real module anatomy: manifest, product entry, RouteMounts, authz, shared API, CSS, functions, tests.
  • Server auth uses makeModuleAuthz: session, enable-gate, Owner bypass, then matrix check.
  • Tenant scope is consistently anchored on client_id or bucket_id.
  • Depth features are real CRUD/approval flows, not static mock screens.
  • Project Service reuses Workforce auth and product enablement instead of inventing a second product shell.
  • Tests cover most handlers and include characterization coverage for branch-matrix simplification.
  • CSS uses dark-theme tokens and .wf-* namespacing.

Bad

  • shared/api.ts is a 674-line mixed type/client file. It is navigable, but not an industry-grade boundary.
  • Permission lists are duplicated in _workforce-authz.ts and WorkforceRouteMounts.tsx; there is no shared/permissions.ts.
  • Many comments narrate obvious behavior. The new rule should reduce this without deleting comments that explain constraints.
  • Some UX still asks users to paste UUIDs for employees, assets, and files. That is acceptable for an internal prototype, not for production HR ops.
  • Validation is manual and inconsistent across handlers; several routes rely on Postgres casts for invalid UUID/date errors.
  • Payroll is explicitly tracking only. It lacks statutory deductions, payslip generation, export formats, and lock/reopen controls.
  • Audit logging is thin for sensitive actions like approval, payroll approval, asset assignment, and training certification.

Missing

  • Employee master profile: employment status, start/end dates, department, work location, job title, manager override, and emergency fields.
  • Scheduling standards: availability, shift conflict prevention, break rules, local holidays, labor-rule policies, and schedule publish/lock.
  • Leave accrual: policy rules, carry-forward, adjustments ledger, holidays, and manager-facing balance history.
  • Time clock rigor: device/kiosk mode, correction requests, manager edits, and immutable punch history.
  • Reporting: utilization, absence rate, overtime cost, training compliance, asset custody, payroll export, and workforce cost by project.
  • Operational safeguards: audit log rows, CSV export, seeded demo coverage for all tabs, real-browser mobile verification evidence.

Frontend Product Roadmap: S1-S8

This is the product roadmap for turning the existing Workforce backend depth into a real workforce frontend. The baseline problem remains: backend depth arrived before product workflow design. The module has the nouns: shifts, leave, punches, payroll, courses, assets, profiles. It still needs manager workflows: publish schedule, detect conflicts, request/approve/correct, notify, export, audit, and guide daily decisions.

Team Integration Contract

  • Team remains the source of truth for users, credentials, access levels, invitations, module permissions, and active/inactive account state.
  • Workforce extends Team; it must not duplicate login identity, role hierarchy, access-level editing, or the Team drag/drop level board.
  • Employee records link to Team through user_nodes where the person has a login, and to booking_resources where the person/resource is schedulable.
  • Workforce can show employment context such as department, manager, hire date, worker type, documents, assets, leave, punches, training, and payroll readiness.
  • Workforce cannot silently mutate access. Role, level, permissions, invite, disable, and credential flows stay in Team/AMS, with deep links from Workforce when needed.
  • Owner bypass and module gating must stay identical to Team, Sidebar, RouteMounts, and requireWorkforce.

Build Constraint

  • Every Workforce employee picker should show Team-backed names, emails, access levels, and employment status, not raw UUID fields.
  • Every sensitive Workforce write should preserve Team boundaries: no cross-client user lookup, no privilege escalation, no hidden level changes.
  • Every Workforce profile page should make Team state visible: invited, active, disabled, access level, products/modules available, and whether the person has login access.
  • When a workflow needs account/access changes, route the user to Team instead of reimplementing Team controls inside Workforce.

X01 Field Lesson: Attendance Time Must Belong to the Workspace

Added 22 July 2026. This is a production-behaviour lesson, not a cosmetic timezone preference. A clock record is an instant, but an attendance day, a shift start, lateness, and an AM/PM label all have meaning only in the client workspace's configured timezone.

X01 Problem: Correct timestamps appeared to have the wrong AM/PM

A senior reviewing an employee timesheet saw clock records such as 10:40 PM to 7:40 AM, even though the employee had worked the expected daytime shift. The database values were not reversed. They were UTC instants displayed by the reviewer's browser in its own timezone instead of the workspace timezone.

  • A value stored as 2026-07-20T05:40:00Z is 11:10 AM in Asia/Kolkata, but can display as 10:40 PM on the previous day in a UTC-7 browser.
  • The old page mixed two incompatible rules: new Date(value).toLocaleTimeString() for display, which follows the viewer's browser, and timestamp.slice(0, 10) for grouping, which follows UTC.
  • That creates contradictory evidence: a row can appear under one calendar day while its displayed time belongs to another local day. It also makes “today”, late detection, date filters, and payroll-period review unreliable around midnight.
  • The manager workflow made diagnosis slower: the action lived under a vaguely named Punching tab, with no direct attendance path from schedule planning or an employee profile.

X01 Decision: UTC storage, workspace-local interpretation

Keep durable event timestamps as PostgreSQL timestamptz values. Do not rewrite historical punches and do not store a second local timestamp. Instead, interpret an instant through the owning client's clients.timezone everywhere a human calendar value is needed.

  • Storage invariant: a punch/event is one absolute instant in UTC. This preserves auditability and lets a record remain unambiguous if a reviewer travels.
  • Workspace invariant: an attendance date, clock label, weekday, late calculation, and range filter use the client timezone, never the reviewer computer timezone.
  • Browser invariant: a Workforce page receives the timezone from authenticated client context; it does not infer it from Intl.DateTimeFormat().resolvedOptions().timeZone.
  • Server invariant: when SQL compares a timestamp to a date or weekday, it converts first: (punched_in_at AT TIME ZONE client_timezone)::date. UTC CURRENT_DATE is not an attendance-day definition.

What We Did to Solve It

Made the workspace timezone available to the browser. u-me.ts now returns client.timezone; the user-portal client type carries that optional value. Login and client-slug responses return it too, so the authenticated client model stays coherent. No new table or migration was needed because clients.timezone already existed.
Created one Workforce time boundary. src/modules/workforce/shared/time.ts validates a supplied IANA timezone, falls back safely to UTC only when it is absent/invalid, and exposes formatWorkforceTime, formatWorkforceDate, and workforceDateKey. Pages no longer each decide how to turn an ISO instant into a clock label or day key.
Corrected manager and employee display paths. Attendance cards, history, ledger events, correction dates, and the employee dashboard's “Today’s time” tile now use the shared formatter with the workspace timezone. The Attendance screen visibly states which timezone it is using, making the interpretation reviewable rather than implicit.
Corrected server-side day boundaries. workforce-punches.ts uses the workspace timezone for from/to filters, duplicate clock-in checks, shift weekday matching, and late-minute calculations. workforce-me-clock-in.ts uses the same rule for self-service lateness; workforce-me-time-status.ts uses it for today's event list. Browser and server therefore agree on what “today” means.
Made attendance discoverable for managers. The former Punching navigation label is now Attendance; Timesheets is now Time entries to distinguish manual records from clock evidence. Staff & Schedule has a Review attendance command, and every linked employee profile has View attendance, which opens Attendance already filtered to that employee.

Why This Fix Is Correct

The system now separates three concepts that were previously conflated:

  1. Instant: when the event happened globally. PostgreSQL owns this as timestamptz.
  2. Business calendar: which workplace day, shift, or pay period the event belongs to. The client timezone owns this.
  3. Viewer locale: language/formatting preferences. This may affect presentation style, but must not change the attendance date or AM/PM meaning.

This is why changing only the React formatter would have been incomplete: server filters and late calculations could still disagree with what the manager sees.

Guardrails for Future Work

  • Never use toISOString().slice(0, 10), timestamp.slice(0, 10), CURRENT_DATE, or EXTRACT(DOW FROM NOW()) for a Workforce business-day decision unless the query has first converted through the client timezone.
  • Never use browser-local toLocaleTimeString() directly for Workforce attendance. Use the shared helper so a travelling manager cannot reinterpret the employee's day.
  • Any new Workforce reporting, payroll close, leave overlap, attendance export, or schedule planner date filter must explicitly identify whether it is an instant or a workspace-calendar value.
  • Do not silently change the workspace timezone after history exists without a product policy. Historical instants remain correct, but their calendar classification can change if the business timezone changes.
  • When a manager needs a person's records, provide an employee-context entry point. A broad tab plus an unlabeled filter is not a complete operational workflow.

X03-X04 Field Lesson: A Correction Must Become an Explainable Decision

Added 22 July 2026. A manager should never have to infer whether an approved change reached payroll, and an employee should never have to infer whether their correction was accepted. This implementation establishes the first end-to-end payable adjustment and correction-review path.

X03 Problem: Payroll could calculate from a changing operational table

Approved manual time entries were used directly when payroll was viewed or exported. A later correction, an imported approved entry, or a future additional source could therefore produce a total without a single durable explanation of the pay decision.

  • Attendance evidence, manual entries, and corrections answer different questions. Treating them as the same record invites silent omission or double counting.
  • A payroll reviewer needs the source record, work date, signed-off actor, approval moment, minutes, and reason for every payable change.
  • Historic/imported approved entries must not disappear merely because they predate a new workflow.

X03 Decision: Payroll reads one immutable payable-time ledger

workforce_payable_time_entries is now the only input to payroll detail, payroll approval, exports, and payslips. It stores a source type and source ID under a unique constraint, so one approved source can create at most one payable line.

  • Approved manual time: approving a Team-linked timesheet creates its ledger entry in the same SQL statement.
  • Approved correction: creates a positive or negative adjustment linked to the correction, not a mutation of original evidence.
  • Reconciliation: payroll reads idempotently capture older/imported approved timesheets before calculation. This repairs legacy drift without calculating directly from the mutable table.

What We Did for X03

Allocated migration 168. 168_workforce_payable_time_reconciliation.sql creates the payable ledger, indexes it by client/date/person and client/resource/date, enforces non-zero bounded minutes, and ensures a client/source-type/source-ID combination is unique. It backfills already approved Team-linked timesheets once.
Moved payroll to the ledger boundary. Payroll period detail, approval totals, CSV export, and payslip generation use computePayableLineItems. Rate selection remains effective-date based, but hours and amount now come only from approved ledger records.
Made timesheet approval atomic. The timesheet update and its payable insert share one SQL CTE. A network failure cannot leave the entry approved while its payable source is missing.
Added reconciliation as recovery, not a second truth. Before a payroll read, approved imported timesheets are inserted into the ledger with ON CONFLICT DO NOTHING. The process is repeatable and cannot add the same source twice.

X04 Problem: A correction could stop at “requested”

A correction request had no structured manager decision, resolution note, payable outcome, or clear audit record that connected the request to payroll. That leaves the employee unsure and creates a manual reconciliation burden for payroll.

  • A denial without an explanation does not close the employee's question.
  • An approval that overwrites the original punch destroys the evidence needed to review the decision later.
  • Separate database writes can result in a correction marked approved with no corresponding pay adjustment.

X04 Decision: Review creates an audited outcome

A manager reviews a pending correction with a required decision note. Approval creates an explicit payable adjustment; denial records the reason and neither path changes the original clock evidence. Both append a ledger event showing who decided and when.

  • The manager supplies a work date and non-zero minutes only when approving, including negative minutes for a valid deduction.
  • The correction stores its payable ledger ID and applied timestamp, so a future reviewer can follow the record without guessing.
  • The review SQL is one operation: state, payable adjustment, and audit event succeed together or fail together.

What We Did for X04

Allocated migration 169. 169_workforce_correction_resolution.sql adds a resolution note, applied timestamp, and a one-to-one link from a correction to its payable entry.
Built a protected review endpoint. PATCH /api/workforce/time-correction/:id accepts only approve or deny from a Workforce manager, requires a decision note, validates date/minutes, rejects a second decision, and distinguishes a missing request from an already reviewed one.
Made the Attendance queue actionable. Pending corrections now expose a Review action. The employee's requested time seeds the manager form where available; the manager still supplies the exact payable date and minutes so the decision is deliberate rather than inferred.
Tested the operational accounting rule. The focused reconciliation test proves one approved timesheet produces one payable source, an approved 90-minute correction becomes 1.5 paid hours at the applicable rate, and a denied correction never reaches payroll.

Why This Boundary Is Correct

  1. Evidence remains evidence: clock records and original manual entries are not rewritten to make a pay figure look tidy.
  2. Approval remains a decision: only an approved source becomes payable, with actor, time, source, and reason retained.
  3. Payroll remains explainable: the calculation starts from a finite ledger rather than reconstructing different operational screens each time.
  4. Recovery remains safe: reconciliation uses source idempotency, so it can repair legacy imports without duplicate pay.

Remaining Scope

  • This is the correction-review slice, not yet the shared manager exception inbox. X08 still needs ownership, due time, delegation, escalation, and cross-system items.
  • Payroll periods still need X10 frozen close snapshots, exception gating, controlled reopen/dispute handling, and a line-by-line payslip explanation.
  • Attendance-only and overtime sources need explicit reconciliation policies before they are added to payable time. They must not be implicitly folded in.
  • Real-browser manager and employee review testing remains required before release, particularly on smaller screens.

X05-X06 Field Lesson: A Schedule Is Not Real Until the Employee Can See It

Added 22 July 2026. Team identity tells the system who is acting. Workforce must use that identity to present the employee's own records and a dated schedule they can rely on. A recurring template is a manager's editing tool, not proof that an employee was notified.

X05 Problem: Self-service could still lose the employee's story

The employee dashboard already derived identity from the signed-in Team user for leave, swaps, and corrections. However, cancelling a pending correction deleted it. The manager had an event trail, but the employee's request history no longer showed that they had withdrawn it.

  • Deleting a high-impact request turns a valid state change into an absence of evidence.
  • A bare approved or denied label does not tell an employee why their correction was decided.
  • Any self-service endpoint that accepts a different resource or user ID would allow the employee surface to become an administrative form.

X05 Decision: The signed-in Team user owns the lifecycle

Self-service resolves the active employee profile from the authenticated Team user and scopes every action to that person. Cancellation is a durable cancelled state, not a delete; manager resolution notes and request status remain visible in the employee dashboard.

  • Leave cancellation, swap cancellation, and correction cancellation are all limited to the employee's own pending record.
  • A correction cancellation writes both the state transition and a time-ledger audit event in one SQL operation.
  • The dashboard presents the manager's resolution note, so a person sees an outcome rather than needing to ask whether the request vanished.

What We Did for X05

Allocated migration 171. 171_workforce_correction_cancellation.sql extends the correction state machine from pending → approved|denied to include pending → cancelled. A completed decision cannot be withdrawn.
Added a scoped cancellation endpoint. DELETE /api/workforce/me/time-correction/:id resolves the signed-in employee, verifies ownership and pending state, retains the row with a cancellation note, and rejects a second cancellation.
Made cancellation auditable and visible. The update and a self-service correction event occur together. The employee dashboard shows the persistent status and manager/cancellation note, while the cancel command is available only while the request remains pending.
Preserved the Team boundary. Neither the dashboard nor the endpoint accepts an employee selector. Team continues to own identity and access; Workforce adds only the employee's operational record.

X06 Problem: Recurring shifts looked like a communicated schedule

workforce_shifts stores weekly repeating templates. If a manager edits one, the old system had no dated record of which shift pattern was shared for a particular week, whether a previous schedule was replaced, or whether an employee acknowledged it.

  • A later template edit could silently change the apparent schedule for a week already under way.
  • An employee could not distinguish an upcoming published shift from an internal manager draft.
  • A manager could not establish a recipient list or measure acknowledgement without manually reconstructing it.

X06 Decision: Templates are drafts; published weeks are immutable snapshots

Publishing materialises a Monday-starting weekly version from recurring shifts. The snapshot stores dated assignments and one in-app notice per Team user. Re-publishing supersedes, rather than overwrites, the prior publication.

  • Managers edit recurring templates, select the target week, decide whether acknowledgement is required, and publish deliberately.
  • Employees see only currently published upcoming snapshot shifts assigned to their Team identity.
  • One published schedule per workspace/week is enforced by a partial unique index; replacement versions preserve the old record as superseded.

What We Did for X06

Allocated migration 170. 170_workforce_schedule_publication.sql adds schedule versions, immutable version-shift snapshots, and per-user schedule notices. The snapshot keeps the source recurring-shift ID only as provenance; it does not depend on a future template edit for its dates or hours.
Added manager publication. POST /api/workforce/schedule-publication validates a Monday week start, rejects an empty template schedule, supersedes the current publication for that week, creates the new version, copies all shifts to dated rows, and creates distinct notices for assigned Team users in one transaction.
Added manager visibility. Staff & Schedule now shows Draft versus Published status, recipient acknowledgement progress, a deliberate acknowledgement requirement, and a clear warning that template changes stay draft until publication.
Added employee acknowledgement. The employee dashboard shows only future published shifts attached to the signed-in Team user. PATCH /api/workforce/me/schedule-notice/:id can acknowledge only that employee's currently published, acknowledgement-required notice.

Why This Boundary Is Correct

  1. Draft is editable: recurring shifts can continue to be managed without rewriting employee-visible history.
  2. Publication is evidence: the dated version states what was communicated for that week and who received it.
  3. Replacement is explicit: a revision supersedes the prior publication rather than altering it in place.
  4. Acknowledgement is scoped: the employee can acknowledge their notice, not another person's schedule or a manager-only draft.

Remaining Scope

  • Notices are in-app records today. Outbound email, push, or SMS delivery and delivery-failure tracking need a deliberate notification integration.

X07 Field Lesson: A Geofence Must Protect Attendance Without Trapping the Employee

Added 22 July 2026. Location is evidence for attendance, not a reason to strand someone outside the system. A legitimate employee can have denied GPS permission, poor accuracy, no signal, or an approved temporary site. The product must retain the rule, explain the result, safely recover from a retry, and route an exception to a person who can decide it.

X07 Problem: A rejected location became a dead end

The first self-service clock-in implementation correctly rejected unassigned, inaccurate, and out-of-radius locations. It recorded the failed event, but a person could only retry. If the device had no location permission or the shift was legitimately at a temporary site, the employee had no structured next step and a manager had no dedicated evidence-led queue.

  • A browser retry after a slow network response could look like a second action even when the first punch had succeeded.
  • “Allow location access” is insufficient guidance when the device cannot produce a reliable location or the configured worksite does not describe the day's work.
  • A manager override without a captured failure reason, geofence result, actor, and decision note would quietly undo the attendance policy.

X07 Decision: Keep automatic geofencing strict; recover through an explicit, audited exception

Automatic clock-in still requires an assigned Workforce work location and a passing server-side distance and accuracy check. When that cannot happen, the employee may submit a reasoned recovery request. A manager can approve a supervisor override only from the queue, with the recorded attempt time and evidence retained beside the decision.

  • Every self-service clock command can carry an idempotency key, so a lost response replays the original result instead of creating another punch or break.
  • Recovery reasons distinguish permission denial, unavailable or timed-out location, poor accuracy, outside-geofence attempts, missing configuration, and network interruption.
  • An approval creates a normal punch plus a linked time-clock event marked as a supervisor override. It does not rewrite, erase, or disguise the original location failure.

What We Did for X07

Allocated migration 172. 172_workforce_attendance_recovery.sql adds command keys to punches, breaks, and time-clock events, with partial unique indexes scoped to the workspace, Team user, action, and key. It also adds the attendance-recovery record: employee, failure code, reason, attempted time, optional location evidence, status, manager decision, and resulting override punch.
Made self-service retries safe. Clock in, clock out, start break, and end break accept an optional idempotency_key. The dashboard creates one per command. A repeated successful clock-in command returns the original punch as a replay rather than creating a duplicate; the ledger also suppresses duplicate evidence for the same command.
Added a human recovery path. The dashboard translates browser permission, timeout, and availability failures into plain language. For a failed geofence check or location-capture failure, the employee can submit POST /api/workforce/me/attendance-recovery with a reason and any captured coordinates, accuracy, nearest worksite, distance, and geofence result.
Added a manager decision queue. Attendance now loads pending recovery requests separately from corrections. PATCH /api/workforce/attendance-recovery/:id requires a decision note. Approval creates a supervisor-override clock-in at the server-recorded attempt time and an event tied to the recovery ID; denial retains the same evidence and outcome.

Why This Boundary Is Correct

  1. Location remains minimum necessary: it is captured only for the attempted attendance command or a recovery that the employee deliberately submits.
  2. Automatic approval remains impossible: a failed geofence is not converted into a punch by the client, a retry, or a generic manager edit.
  3. The manager decides from evidence: the queue identifies the employee, failure category, reason, attempt time, nearest worksite, distance, and accuracy before an override is considered.
  4. The historical record stays explainable: the original failed event, recovery request, manager note, override punch, and linked event form one traceable story for payroll and audit review.

Remaining Scope

  • The current dashboard requires an online request submission. Durable browser-side offline storage and automatic background replay need a deliberate client queue, conflict policy, and device-security review.
  • Temporary worksites still use a supervisor override. A future location-assignment workflow can add time-bounded, schedule-linked temporary-site eligibility before a shift begins.
  • Recovery approval now follows the X08 policy owner and delegation model. Automatic overdue escalation, reminder delivery, and a substitute owner after a missed response target remain future work.
  • Real-device checks are still required for browser permission prompts, low-accuracy GPS behaviour, mobile retry timing, and the wording shown in each platform's location error.

X08 Field Lesson: A Pending Request Is Work Only When Someone Owns It

Added 22 July 2026. A request that merely says pending is not operationally managed. Leave, overtime, shift swaps, time corrections, attendance recovery, and payroll can all wait indefinitely when a manager is absent or the decision rule lives only in a person's memory. Workforce now assigns an accountable decision owner, makes the due point visible, and permits narrowly scoped cover without changing Team access.

X08 Problem: Approval authority was scattered and implicit

Each Workforce screen had its own approve or deny action. The action checked a broad module permission, but it did not consistently answer who was responsible for this request, who could act while that person was away, or when the decision was due.

  • A manager could spend time searching Leave, Overtime, Punching, Swaps, and Payroll to find requests that were waiting.
  • Delegating coverage in Team would have changed someone's workspace access, which is much broader than granting a temporary approval duty.
  • A request with no visible owner or response target becomes a silent service failure rather than an exception that can be managed.

X08 Decision: Route every decision type through policy ownership

Each request type has a policy with a primary Team user and a response target in hours. When a policy intentionally has no primary owner, the employee profile's recorded manager is the compatibility fallback. A temporary delegation is specific to one request type, one owner, one delegate, and an optional expiry. It grants approval responsibility only; it does not modify Team levels, roles, login access, or wider Workforce permissions.

  • The Level 1 Owner remains an explicit operational override, preserving the existing owner bypass rather than leaving an absent manager as a single point of failure.
  • Decision endpoints enforce the same routing rule, so the inbox is not just a visual suggestion.
  • Delegation, revocation, policy changes, and final decisions leave a routing event for later audit review.

X09 Field Lesson: Workforce Access Is Not Permission to Read Every Employee Detail

Added 22 July 2026. Workforce records combine ordinary operating data with personal contact details, emergency contacts, compensation, and precise location evidence. A broad module-view permission is enough to run a schedule; it is not enough to expose every sensitive field to every colleague. The system must separate Team membership from data purpose and leave evidence when sensitive data is accessed.

X09 Problem: Broad Workforce views silently exposed sensitive records

Employee directory and master-profile APIs returned personal phone, non-login email, emergency contact, and arbitrary custom fields to any caller who could view Workforce. Payroll rates were similarly tied only to the Payroll screen. Clock evidence and recovery queues could return exact latitude, longitude, accuracy, and distance through otherwise general manager workflows.

  • A Team role answers what a person can do in the workspace; it does not prove a continuing need to know somebody's pay, emergency contact, or location trail.
  • Hiding a field in React is not a privacy control when the API already sent the value to the browser.
  • Location data is particularly easy to overcollect: a manager may need a pass/fail result for attendance, not the employee's exact coordinate history.

X09 Decision: Use named, purpose-limited data scopes

Workforce now recognises three sensitive scopes: profile for personal contact and emergency details, compensation for pay-rate records, and location_history for precise worksite and clock evidence. The Level 1 Owner retains the documented override. An Owner may grant one scope to one Team user with a business reason, and may revoke it without changing the person's Team role or module permissions.

  • A recorded direct manager may see their direct report's profile-contact fields without a global profile grant; all other profile viewers receive redacted values.
  • Exact location is opt-in on manager history endpoints. The normal attendance queue still shows the operational result and reason, but not coordinate evidence.
  • Grant-based and Owner-based sensitive reads write a small access event containing the actor, scope, endpoint, subject where known, and access basis. The event never stores the sensitive value itself.

What We Did for X09

Allocated migration 174. 174_workforce_sensitive_data_access.sql adds a workspace-scoped sensitive-data grant register and immutable access-event table. One user can hold at most one grant per scope; updating a grant reactivates or revokes that named capability rather than creating an ambiguous history.
Separated data access from Team access. GET|POST /api/workforce/sensitive-access is Owner-only. The Privacy tab presents the three scopes, asks for a Team user and business reason, lists active and revoked grants, and revokes future access immediately. This does not modify the Team hierarchy, role, login, or ordinary Workforce permissions.
Redacted profile APIs by default. Employee directory and master-profile responses now withhold personal phone, non-login email, emergency contact, and custom fields unless the caller is the Owner, has an active profile grant, or is that employee's recorded direct manager. Profile writes from an unapproved actor retain existing sensitive values instead of replacing them with blank form data.
Scoped pay and location evidence. Payroll-rate reads and writes require the compensation scope. Worksite configuration requires the location-evidence scope. Generic time-ledger and attendance-recovery queues omit exact coordinates unless include_location=true is requested by an Owner or a user with an active location-evidence grant; decision responses apply the same redaction.

Operational Flow

  1. Start with the least scope: managers use ordinary Workforce screens for schedules, attendance outcomes, and approvals. They do not automatically see personal profile, pay, or precise GPS data.
  2. Grant for a defined responsibility: the Owner selects the minimum scope and records why, for example temporary payroll cover or an HR emergency-contact duty.
  3. Use location evidence deliberately: a manager reviewing a failed attendance request normally sees the failure and employee reason. Exact coordinates are requested only when they are necessary to resolve the case.
  4. Revoke at the end of cover: the Owner removes the grant when the business need ends. Future reads are blocked; access evidence remains available for audit.

Remaining Scope

  • Direct-manager profile access is based on the recorded manager relationship. Acting manager, HR caseworker, and emergency-access workflows need time-bound, case-linked rules before they can bypass that relationship.
  • Access events are retained for audit but do not yet have a manager-facing audit explorer, export retention policy, or automated unusual-access alert.
  • Data export, files, and other modules have their own privacy surfaces. A whole-product data-retention and subject-access-request programme remains outside this Workforce migration.
  • Real-browser review is still required for the redacted employee form, the Owner-only Privacy tab, revoked-grant refresh, and a manager attempting the location-evidence view without a grant.

X10 Field Lesson: Closing Payroll Must Freeze the Explanation, Not Just the Total

Added 22 July 2026. A payroll period can look approved while still being calculated from live payable time and live rates. That means a legitimate later correction or rate change can silently change a period that a manager already reviewed. The operational failure is not merely a wrong number: payroll staff can no longer explain why an old export changed, and a worker has no durable record to dispute.

X10 Problem: Exporting live data rewrote closed payroll

The original period screen recomputed payable entries and rates whenever it was read. Export generation used the same live calculation and upserted the existing payslip for the employee and period. A changed hourly rate, newly approved correction, or late source record could therefore alter a previously generated amount without creating a new version or explaining the difference.

  • An approval total is not evidence if its inputs remain live and mutable.
  • A payslip upsert can make an old export appear to have said something it did not say when it was generated.
  • Asking payroll staff to remember the old rate or reconstruct the source rows from several screens is not an audit trail.

X10 Decision: Approval creates one immutable calculation snapshot

Approving a draft period now creates a workspace-scoped snapshot before the period changes to Approved. Each worker line freezes total hours, blended hourly rate, gross and net values, currency, and the individual payable ledger sources with the rate used for each source. The approved period points to that one snapshot; all later reads, exports, and payslips use it instead of recalculating live history.

  • Rate changes and later payable entries affect only future draft periods; they cannot rewrite a frozen snapshot.
  • Export is idempotent per snapshot. Repeating the command returns the existing export and payslips rather than creating a new amount or updating the old payslip.
  • A dispute is a separate, status-tracked record linked to the period, snapshot, optional payslip, and affected Team user. Resolving a dispute records the decision without changing the closed evidence.

What We Did for X10

Allocated migration 175. 175_workforce_payroll_snapshots_disputes.sql adds one snapshot per workspace payroll period, immutable snapshot lines, snapshot references on payroll periods, exports, and payslips, and a durable payroll-dispute register. Snapshot lines retain source evidence rather than only storing a final total.
Changed the close boundary. PATCH /api/workforce/payroll/:id now reconciles approved time, calculates the line evidence once, persists the snapshot, then marks the period approved with the frozen total and snapshot reference. A period GET returns frozen lines after close and a clearly live preview while the period is still draft.
Changed export semantics. POST /api/workforce/payroll-export rejects a draft or snapshot-less period. For a frozen snapshot it creates one export and one immutable payslip per snapshot line. A repeat request returns that existing batch. Legacy payslips without a snapshot are blocked from silent overwrite and must be deliberately voided or migrated by payroll operations.
Added a controlled discrepancy path. GET|POST /api/workforce/payroll-disputes lists and records disputes. PATCH /api/workforce/payroll-dispute/:id moves an open case through review, resolution, or rejection with a required decision note at close. The Payroll screen labels Draft calculations as live previews, labels approved calculations as Frozen, exposes the export register, and provides a dispute register for the selected frozen period.
Protected the data purpose. Period detail, export, and dispute endpoints require both the existing Payroll permission bucket and the compensation sensitive-data scope introduced in X09. Reading compensation evidence writes the same purpose-limited access event instead of treating a broad Workforce view as entitlement to payroll detail.

Operational Flow

  1. Prepare: payroll works in a Draft period, where time and rates are visibly a live preview. Missing or incorrect source time is fixed through the normal time and correction workflows before close.
  2. Close: the assigned payroll approver approves the period. Workforce freezes each included payable source and rate into one snapshot and stores the closed total.
  3. Export: payroll generates the provider file or review batch from that snapshot. Repeating the action is safe: the existing batch is returned, not recomputed.
  4. Question: payroll records a dispute against the frozen employee line or payslip. The reviewer investigates the preserved evidence, records a review outcome, and leaves the original period intact.
  5. Correct forward: once a future-period adjustment policy is approved, the resolution should create a separately identified adjustment in a future draft period. It must reference this dispute and never edit the original snapshot.

Remaining Scope

  • Disputes are durable and reviewable, but automatic carry-forward adjustment creation is deliberately not inferred. It needs an approved adjustment type, target-period selection, accounting treatment, and provider export rules.
  • Snapshot lines preserve time and rate evidence. Tax, deductions, leave valuation, overtime multipliers, statutory calculations, and provider-specific payslip layout remain outside the current tracking-only payroll scope.
  • Void and replacement-export controls for legacy unsnapshotted payslips need a payroll-operations UI and a documented migration procedure before they are exposed as a routine action.
  • Real-browser verification remains required for a compensation-granted payroll reviewer, a repeat export, a rate change after approval, and a dispute resolved through the manager workflow.

X02-X10 Field Lessons: Workforce Must Be Operated, Not Merely Configured

Added 22 July 2026. A Workforce product succeeds when a person on a busy shift can complete the next right action and a manager can resolve the exceptions without reconstructing the story from separate tabs. The roadmap therefore has to cover operational ownership, recovery, communication, and trust alongside schemas and endpoints.

The Operational Failure Mode

A tab can be technically complete and still fail in daily use. The current module exposes many capable screens, but a user can still be forced to decide where a task belongs, select the person they already are, discover prerequisites only after an action fails, or manually join attendance, corrections, schedules, and pay records.

  • An employee should not need to understand the Workforce information architecture to clock in, request leave, or see the result of a request.
  • A manager should not have to visit every tab to discover missed punches, under-coverage, pending leave, expiring training, overdue assets, and payroll blockers.
  • An Owner needs clear setup order and policy controls, not a collection of forms whose dependencies are implicit.
  • Payroll and audit reviewers need a frozen, explainable account of what was approved, including corrections and overrides, rather than a total that can change after the fact.

The Product Decision

Design Workforce around three operating surfaces and a shared exception model. The same Team identity remains the source for access, while Workforce extends that person only with employment and operational data.

  • Employee surface: today's shift and time status, personal requests, personal balances, assigned learning/assets, and a clear recovery path when something goes wrong.
  • Manager surface: an action-ranked daily inbox for coverage, attendance, approvals, corrections, readiness, and follow-up; every item has an owner, due point, state, and resolution.
  • Owner/payroll surface: guided configuration, policy visibility, publishing controls, period close, audit exports, and controlled reopening.
  • Shared rule: the user sees the task in the context where they need it, while detailed records remain reachable through direct links rather than hidden filters.

The Daily Operating Loop

Employee begins work. The dashboard identifies the employee automatically, shows today's scheduled work and allowed worksite, and provides Clock in, Break, Clock out, or a clearly named fallback such as Request correction. It never asks the employee to select themselves from a staff list.
Manager runs the day. A single review queue surfaces late/missing punches, failed location checks, uncovered or skill-ineligible shifts, leave awaiting a decision, swaps, expiring certificates, and assets due back. The manager can open each item already scoped to the relevant person, shift, or request.
Owner publishes and controls policy. Setup has an explicit order: Team person, employment profile, worksite/eligibility, schedule, publish. Policy changes state their effective date and never silently reinterpret past attendance or payroll.
Payroll closes evidence. The period cannot become payable until reconciliation is complete. Closing produces immutable line items and a human-readable explanation; a later correction follows a controlled reopen/dispute path rather than silently changing a past payslip.

Acceptance Questions for Every Workflow

  • Who performs this action in normal work: employee, manager, delegated manager, owner, payroll reviewer, or system?
  • What starts it, what information is already known, and what is the shortest valid path?
  • What happens when the network, GPS, policy, schedule, or approver is unavailable?
  • Who owns the next decision, when is it due, how is it escalated, and how does the requester see the outcome?
  • Which downstream records change, and how can a future reviewer explain that change without reconstructing hidden state?

Implementation Guardrails

  • No silent background transition for a high-risk Workforce state. The user needs confirmation, record state, and a route to evidence.
  • No generic empty state where a prerequisite is missing. Explain the blocked condition, the responsible role, and the next action.
  • No manager override without a reason, actor, time, and reviewable audit event.
  • No workflow is complete because its create endpoint exists. It needs resolution, cancellation, notification, visibility, permission, and failure-path coverage.
  • No dashboard counter without an actionable drill-down that preserves the relevant person/date/request context.
