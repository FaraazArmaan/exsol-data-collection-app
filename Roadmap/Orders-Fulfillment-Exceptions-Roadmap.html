ExSol / Product Architecture Field Guide

Orders fulfillment exceptions: finish the customer promise without rewriting history

A first-principles learning guide for one difficult case: some units were fulfilled, the customer cancels the rest, stock is released only where it is still held, and Payments—not Orders—proves any refund.

Recommendation: keep “Partially fulfilled” as a line-derived Orders label. Persist only the cancellation facts that cannot be reconstructed safely: who cancelled which remaining line quantity, why, the exact refund amount snapshot, and the linked Orders refund request. Do not add a new global sale_status.

Scope

One exception

A paid pickup/online sale has fulfilled quantity on at least one line, then its remaining, unfulfilled quantity is cancelled.

Ownership

Orders decides goods

Orders owns fulfillment/cancellation facts and tells Inventory to release the unused hold. Payments owns provider calls and money evidence.

Out of scope

No Procurement or catalog redo

No purchase orders, receiving flow, product roadmap, stock policy redesign, deployment, or migration-number allocation.

2. Domain decision: quantities are facts, labels are derived

Chosen rule

Line arithmetic

For every sale_line:

ordered_qty   = sale_lines.qty
fulfilled_qty = Σ fulfillment_line.qty where fulfillment.status = fulfilled
cancelled_qty = Σ persisted cancellation-line.qty
remaining_qty = ordered_qty − fulfilled_qty − cancelled_qty

invariant: 0 ≤ fulfilled_qty + cancelled_qty ≤ ordered_qty

“Partially fulfilled” is displayed when fulfilled_qty > 0 && remaining_qty > 0. “Remaining cancelled” is displayed when cancelled_qty > 0 && remaining_qty = 0. This works per line and then rolls up to the sale.

Why not a new sale status?

A new sale_status would make a global POS lifecycle field encode an Orders-only mixed outcome. It would complicate POS FSM, Payments’ full-refund rule, reporting, and every existing query.

Derived disposition is sufficient: Orders APIs return the aggregate and Orders action guards consult it. A sale can remain paid while a partial refund is pending or completed.

When persistence is necessary

The cancellation itself is not a label. It must survive changing product data, prove the cancellation amount/reason/actor, prevent a second cancellation, link exactly one refund request, and exclude the quantity from later fulfillment.

Therefore a small Orders migration is needed; no number is selected here.

Cancellation semantics

  • The operator cancels all remaining quantity selected for each line; the server recomputes that remaining quantity and never trusts a browser total.
  • Only quantities in pending, picked, or packed fulfillment work are eligible. shipped, fulfilled, and already-cancelled quantity are rejected with a conflict.
  • A multi-line fulfillment can continue for its other eligible quantities. If all of its effective quantities are cancelled, its header becomes cancelled.
  • Cancellation is idempotency-protected by durable cancellation facts and row locking: a repeat sees no remaining quantity and cannot create another release or refund.
Stock rule: release only the unconsumed reservation: reservation.qty − reservation.qty_consumed. Never add stock back for fulfilled/shipped units and never write a compensating positive sale movement. A later customer return is a separate Inventory returns workflow with its own physical inspection/disposition—not a cancellation.

Orders → Payments refund handoff (the money boundary)

Orders cancellation transactioncreates orders_refunds(requested)
→
Payments submit endpointcreates pending payment evidence + calls provider
→
Signed Payments webhooksettles success/failure
  1. Orders only creates the refund intent. It supplies orders_refund_id, tenant-scoped sale ID, immutable amount snapshot, and a cancellation reference. It does not import a provider SDK/helper or call a provider.
  2. Payments executes. A new Payments-owned authenticated endpoint validates the refund is tenant-owned, linked to a captured allocation, not already submitted, and within the ledger’s pending/completed cap. It writes the pending payment_transactions.provider_refunded row, moves the Orders refund to approved, then makes the Razorpay call.
  3. Payments confirms. Only payments-razorpay-webhook, after raw-body signature verification, amount/currency checks, and event deduplication, writes succeeded and moves the refund to completed; a provider failure returns the refund to requested.

Orders must never mark a refund completed, fabricate payment evidence, or execute the Razorpay request. Existing orders-refund-advance provider behavior must be moved behind the Payments endpoint as part of the approved implementation.

Permission rule

Use existing buckets only—no action-namespaced keys.

  • Cancellation and line visibility: orders.business.edit/view.
  • Provider submission/retry: payments.customers.edit.
  • The Orders UI shows the money action only with both the Orders context and the Payments permission. Missing Payments permission leaves a truthful “Refund requested—Payments permission required” state.
  • Both Orders and Payments use their established enable-gate → L1 Owner bypass → matrix check.

Audit events

  • orders.fulfillment.remaining_cancelled: sale, cancellation ID, line quantities, reason, reservation release count, refund request ID/amount—never provider secret/payload.
  • orders.fulfillment.cancel_rejected only for meaningful operator-visible conflicts if the project’s audit volume policy permits; normal validation failures stay request logs.
  • payments.orders_refund.submitted, payments.orders_refund.provider_outcome_unknown, and webhook settlement/failure evidence are owned by Payments.

6. Storage and migration decision

A forward-only migration is required, but its number must be allocated by the human coordinator after roadmap approval. No number is proposed here.

Recommended additive schemaReason
orders_fulfillment_cancellations
id, client_id, sale_id, reason, created_by, created_at, idempotency_key		Durable operator decision, tenant scope, grouping and replay protection.
orders_fulfillment_cancellation_lines
cancellation_id, sale_line_id, qty, refund_amount_cents		Immutable per-line cancellation and money snapshot. Database constraints: positive quantity/amount, tenant-safe FKs, one cancellation line per sale line for this “cancel all remaining” exception.
Nullable cancellation_id / unique link on orders_refundsOne cancellation produces one traceable refund request; prevents an accidental second automatic refund for the same cancellation.

Do not mutate sale_lines.qty, original price snapshots, existing negative stock_movements, or the captured payment transaction. The cancellation tables are an additive event-like fact layer; derived queries combine them with fulfilled fulfillment lines.

Deterministic refund amount

Use the stored sale-line monetary snapshot—not today’s Product Manager price. The handler computes each line’s cancellation share from the line’s stored, customer-facing amount and records the final integer-cent result. If tax/discount is not fully represented in the line snapshot, stop at an explicit server-side allocation helper with a documented remainder rule (last cancelled unit receives the accumulated remainder); do not accept free-form refund cents for this exception. This needs a characterization check against the current POS quote/create snapshot before implementation.

User journey through those files: a staff member opens the fulfillment detail (FulfillmentsTab), sees the read-model totals from orders-sale-lines, and confirms cancellation. orders-cancel-remaining authenticates with _orders-authz, locks and writes Orders cancellation facts, invokes the Inventory-owned release seam within its database transaction, records an Orders refund request, and returns the updated read model. A separately authorized staff action reaches the Payments endpoint; only that endpoint and the signed webhook handle the provider and financial evidence.

Unresolved approval points

  • Confirm whether a cancelled remainder without a Razorpay capture should stay refund requested—manual finance review (recommended for this scope) or needs a future cash-refund workflow.
  • Confirm the exact stored amount field that represents customer-refundable line value when cart-level tax/discount exists; implementation must not guess.
  • Confirm that cancellation is prohibited once a fulfillment is shipped. Recommended: yes; a later return handles physical stock separately.
