1 · Plain-English purpose

POS sells; Orders keeps the promise

POS and the storefront are the front counter: they price a basket and create a sale. Orders starts after that moment. It explains what must be picked, what was handed over, what could not be supplied, what was cancelled, and which refund is waiting for Payments evidence.

Orders owns

Post-sale operational facts: fulfilment groups/lines, shipment and pickup work, backorders, cancellation facts, SLA events, and future return-case authorisation.

Orders does not own

Catalog facts, live stock, payment evidence, customer master data, or checkout. It references the owner rather than copying mutable truth.

The full promise

Sale created → payment outcome → reservation → fulfilment → pickup/shipping → delivery → cancellation/return → refund/reconciliation.

Design rule. A sale is commercial history. An order operation is work around that sale. “Partially fulfilled” can be derived from line quantities; it is not a new global sale status.

5 · Separate state machines

Do not overload sale_status

A sale can be paid while one box is shipped, another cancelled, a return inspected and a refund pending. One status would lie or lose history. The UI combines owner projections.

Never reverse shipped stock because a remainder is cancelled. Shipped/consumed quantity is historical physical evidence. Only reserved − consumed can release. A return is new receipt/disposition, not negative fulfilment.
6 · Click-to-database architecture

Journeys across boundaries

Paid sale → split delivery

POS/Storefront creates server-priced sale and payment state.
Inventory creates tenant-scoped reservation.
Orders splits fulfilment lines and advances work.
Completion consumes reservation/writes movement; Orders records fulfilment.
Shipment/pickup projection, audit and tests explain result.

Partial fulfilment → cancellation → refund

Orders derives fulfilled and remaining quantity per line.
Staff cancels remainder with reason/idempotency key.
Inventory releases only unconsumed hold; shipped movement stays untouched.
Orders stores cancellation and linked refund intent.
Payments submits/settles; UI shows pending, failed or completed.

Customer return → resolution

POS/Ecommerce creates return request, not stock record.
Orders authorises/refuses canonical return case and instructions.
Inventory receives, inspects and chooses disposition.
Orders requests policy-permitted refund intent.
Payments maps original allocation and proves outcome.

Backorder / provider recovery

Orders records customer commitment.
Inventory/Procurement publish supply facts; never change Orders state directly.
Coordinator resumes only after valid reservation.
Known-safe retries are idempotent; unknown outcome remains visibly pending.

9 · Desktop and phone workflow maps

One record set, two working shapes

Desktop: operations workspace

Queue: visible search, filters, sort, saved views and SLA/exception chips. Detail: sale snapshot, customer reference, line quantities, fulfilments, shipment/pickup, refund/return projection and timeline. Action: one primary next action; destructive cancellation names remaining quantity.

Skeleton, refreshing/stale, no-results/no-orders, denied, conflict and retry must be distinct.

Phone: focused field task

Home: ordered pick/pack/ready/exception queue. Task: 44px actions, scan/touch checklist, short readiness check, reason and sync state. Detail: same history/permission; complex authorisation becomes a full screen.

No implicit offline stock/refund action. Supported local work reconciles centrally.

Finish the narrow safe slice before broadening

Checkpoint fulfilment exceptions

  • Dependencies: 04457b0 reservation semantics and Payments refund contract.
  • Schema: no new migration; 167 is allocated/applied.
  • Acceptance: release only remaining hold, preserve shipped quantity, one refund intent, safe duplicate/cross-tenant/permission/audit behavior and pending/failed/completed UI.
  • Checks: concurrency/idempotency, browser desktop/phone recovery, typecheck/full suite.

Order queue and detail projection

Create a typed tenant-scoped read view before further transitions.

  • Dependencies: POS/Storefront sale read and CRM/Payments summaries.
  • Likely files: Orders page/components, read handler/helper, shared types/API/tests.
  • Schema: preferably none; derive first.

Pickup and shipment evidence

Separate physical handoff from payment. Add idempotent proof only after collector, timestamp, carrier, privacy and exception policy are decided.

  • Dependencies: Warehouse execution decision and Email notification contract.
  • Schema: likely; human allocation required.

Canonical returns

Create Orders return_case_id/lines. Inventory owns receipt/inspection/disposition; Payments owns allocation/refund state; POS/Ecommerce initiates and displays authorised projection.

  • Dependencies: POS/Ecommerce initiation, Inventory projection, Files/CRM privacy and Payments contract.
  • Schema: required across owners; no number allocated.

Warehouse and replenishment collaboration

After Warehouse publishes task contract, Orders may request tasks/consume results. Procurement/Supply Chain publish supply facts, never Orders state.

  • Dependencies: actual Warehouse, integrated Inventory seam and Procurement receipt foundation.

Scale and observability

Saved views, bulk safe actions, scoped exports, alerts, notifications and reconciliation metrics follow stable cases/evidence.

13 · Glossary and learning path

Learn nouns, then trace one record

Sale / order

Sale is commercial agreement. Order is post-sale operational promise.

Reservation / movement

Reservation holds stock; movement changes ledger quantity. Release is not reversing a shipment.

Fulfilment / shipment

Fulfilment prepares/completes items. Shipment or pickup records a handoff.

Backorder / cancellation

Backorder keeps commitment open; cancellation ends only unfulfilled commitment and may create refund intent.

Return / disposition

Return case is service decision; disposition decides physical outcome.

Allocation / idempotency / projection

Allocation links money to obligation. Idempotency makes retry safe. Projection is read model, not competing truth.
